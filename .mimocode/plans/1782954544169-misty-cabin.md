### 1. 布料参数防抖（motion-popup.ts）

**问题**：`motion-popup.ts:370-492` 每个滑块 `onChange` 都同步调用 `recreateCloth()`。`recreateCloth`（`cloth-manager.ts:107-113`）先 `_destroyClothForFocusedModel` 再 `_createClothForFocusedModel`，每步都涉及碰撞器重建、布料粒子网格创建、Mesh 注册。快速拖动时每秒触发数十次完整重建。

**修改方案**：

文件：`frontend/src/menus/motion-popup.ts`

1. 在文件顶部添加防抖工具函数：
```ts
function debounce(fn: () => void, ms: number): () => void {
    let timer: ReturnType<typeof setTimeout> | null = null;
    return () => {
        if (timer) clearTimeout(timer);
        timer = setTimeout(() => { timer = null; fn(); }, ms);
    });
}
```

2. 在 `buildClothParamsLevel()` 函数内（约 L352），创建一个共享的防抖重建函数：
```ts
const debouncedRecreate = debounce(() => recreateCloth(), 100);
```

3. 将所有 `recreateCloth()` 调用（9处，L370/383/396/418/431/444/457/479/492）替换为 `debouncedRecreate()`。

**潜在影响**：
- 滑块拖动期间不再重建布料，松手后 100ms 触发一次重建
- `setEnvState` 仍然同步执行（只更新配置对象），不阻塞
- 低风险：只改变重建时机，不改变最终状态

**验证**：拖动布料参数滑块，确认松手后参数生效；快速连续拖动不会卡顿

### 3. BPM 量化（beat-detector.ts）

**问题**：`beat-detector.ts:138` `this.currentBpm = Math.round(60000 / avgInterval)` 输出原始值（如 118、123），程序化动作直接使用此值计算 `beatFrames`（`procedural-motion.ts:304`），导致节奏偏移。

**修改方案**：

文件：`frontend/src/motion/beat-detector.ts`

1. 添加 BPM 量化辅助函数（纯函数，文件顶部或类内 static）：
```ts
const COMMON_BPMS = [80, 90, 100, 110, 120, 130, 140, 150, 160, 170, 180];
const QUANTIZE_THRESHOLD = 5; // 偏差 ±5 BPM 内量化

function quantizeBpm(raw: number): number {
    for (const bpm of COMMON_BPMS) {
        if (Math.abs(raw - bpm) <= QUANTIZE_THRESHOLD) {
            return bpm;
        }
    }
    return raw; // 偏差大则保留原始值
}
```

2. 在 `update()` 方法 L138 处，将 BPM 量化后赋值：
```ts
this.currentBpm = quantizeBpm(Math.round(60000 / avgInterval));
```

3. 同时更新 `phaseInterval` 以匹配量化后的 BPM：
```ts
this.phaseInterval = 60000 / this.currentBpm; // 量化后的精确间隔
```

**潜在影响**：
- 量化只影响偏差 ±5 BPM 的情况，大偏差（如 110 vs 120）不量化
- `phaseInterval` 与 `currentBpm` 保持一致，`getBeatPhase()` 相位计算更准确
- 程序化动作的 `beatFrames`（`Math.round(((60 / clampedBpm) * 30) / safeSpeed)`）在量化 BPM 下得到更整齐的帧数

**验证**：播放 120 BPM 音乐，确认检测到的 BPM 为 120（而非 118/122）；播放 117 BPM 音乐，确认量化到 120

### 4. VMD 插值曲线（vmd-writer.ts）

**问题**：`buildBoneFrame`（L66-98）插值部分硬编码为线性 `20,20,107,107`（L91-96），所有骨骼用同一条直线插值，动作机械感强。

VMD 插值格式：64 字节，16 组 × 4 字节。每组 4 字节含义：
- `[0]` = X1（入曲线 x，0-127）
- `[1]` = Y1（入曲线 y，0-127）
- `[2]` = X2（出曲线 x，0-127）
- `[3]` = Y2（出曲线 y，0-127）

线性 = `(20,20,107,107)`，缓入缓出 ≈ `(20,40,80,107)`，快入慢出 ≈ `(20,80,107,107)`。

**修改方案**：

文件：`frontend/src/motion/vmd-writer.ts`

1. 定义插值曲线预设：
```ts
export interface InterpCurve {
    x1: number; y1: number;
    x2: number; y2: number;
}

export const INTERP_LINEAR: InterpCurve = { x1: 20, y1: 20, x2: 107, y2: 107 };
export const INTERP_EASE_IN_OUT: InterpCurve = { x1: 20, y1: 40, x2: 80, y2: 107 };
export const INTERP_EASE_OUT: InterpCurve = { x1: 20, y1: 80, x2: 107, y2: 107 };
export const INTERP_EASE_IN: InterpCurve = { x1: 20, y1: 20, x2: 40, y2: 107 };
export const INTERP_SHARP: InterpCurve = { x1: 30, y1: 10, x2: 90, y2: 107 };
```

2. 扩展 `BoneKeyFrame` 接口：
```ts
export interface BoneKeyFrame {
    name: string;
    frame: number;
    position: [number, number, number];
    rotation: [number, number, number, number];
    interp?: InterpCurve; // 新增，可选，默认 LINEAR
}
```

3. 修改 `buildBoneFrame` L91-96，读取 `frame.interp`：
```ts
const interp = frame.interp ?? INTERP_LINEAR;
for (let i = 0; i < 16; i++) {
    view.setUint8(off++, interp.x1);
    view.setUint8(off++, interp.y1);
    view.setUint8(off++, interp.x2);
    view.setUint8(off++, interp.y2);
}
```

4. 在 `procedural-motion.ts` 中为不同骨骼类型指定不同曲线：
- 呼吸/肩膀/手腕 → `INTERP_EASE_IN_OUT`（柔和过渡）
- 手臂摆动 → `INTERP_EASE_OUT`（快速启动慢停）
- 中心弹跳/臀部扭转 → `INTERP_SHARP`（锐利节拍感）
- Idle 模式全部用 `INTERP_EASE_IN_OUT`

**潜在影响**：
- `interp` 字段可选，向后兼容所有现有调用（默认 LINEAR）
- 新增约 5 个常量 + 1 个接口字段，代码量增加极少
- 不影响 MorphKeyFrame（morph 无插值概念，VMD 格式中 morph 帧不含插值数据）

**验证**：生成 Auto Dance VMD 并播放，对比线性 vs 曲线插值的动作流畅度差异

### 5. VMD 动作过渡混合（scene-vmd.ts）

**问题**：`loadVMDMotion`（L86-89）直接 `setRuntimeAnimation(null)` 再设置新动画，瞬间跳变。

**修改方案**：

文件：`frontend/src/scene/scene-vmd.ts`

核心思路：babylon-mmd 的 `MmdRuntime.playAnimation()` 不直接支持 cross-fade。需要利用 Babylon.js 原生的 `AnimationGroup` 混合能力，或者在新旧 VMD 之间做手动权重过渡。

**方案 A（推荐）：手动权重过渡**

1. 在 `scene-vmd.ts` 添加过渡状态管理：
```ts
interface CrossFadeState {
    active: boolean;
    oldAnimation: any;      // 旧 runtime animation handle
    oldWeight: number;
    newAnimation: any;      // 新 runtime animation handle
    newWeight: number;
    progress: number;       // 0..1
    durationMs: number;
    startTime: number;
}

let crossFade: CrossFadeState | null = null;
const DEFAULT_FADE_MS = 500; // 500ms 过渡
```

2. 修改 `loadVMDMotion` 的动画绑定部分（L86-99），创建过渡：
```ts
// 旧动画
const oldHandle = inst.mmdModel.getRuntimeAnimation?.();

// 创建新动画
inst.mmdModel.setRuntimeAnimation(null);
const handle = inst.mmdModel.createRuntimeAnimation(wasmAnimation);

if (oldHandle && !isProcVmdActive()) {
    // 启动 cross-fade
    crossFade = {
        active: true,
        oldAnimation: oldHandle,
        oldWeight: 1,
        newAnimation: handle,
        newWeight: 0,
        progress: 0,
        durationMs: DEFAULT_FADE_MS,
        startTime: performance.now(),
    };
    inst.mmdModel.setRuntimeAnimation(handle); // 先绑定新的
} else {
    inst.mmdModel.setRuntimeAnimation(handle);
}
```

3. 在渲染循环（`scene.ts` 的 `onBeforeRenderObservable`）中更新过渡权重：
```ts
if (crossFade?.active) {
    const elapsed = performance.now() - crossFade.startTime;
    crossFade.progress = Math.min(1, elapsed / crossFade.durationMs);
    // ease-in-out 曲线
    const t = crossFade.progress < 0.5
        ? 2 * crossFade.progress * crossFade.progress
        : 1 - Math.pow(-2 * crossFade.progress + 2, 2) / 2;
    // babylon-mmd 不直接支持 per-handle weight，需要在 WASM 层操作
    // 备选：用 Babylon 的 animationWeight 属性
    if (crossFade.progress >= 1) {
        crossFade = null; // 过渡完成
    }
}
```

**方案 B（简化）：淡出旧动画 → 淡入新动画**

在旧动画上设置 `animationWeight` 从 1→0（约 200ms），然后替换为新动画并设 weight 从 0→1。

**潜在影响**：
- babylon-mmd 的 `MmdWasmAnimation` 是否暴露 per-handle weight 控制需要验证
- 如果 WASM 层不支持权重混合，退化为方案 B（先淡出旧的再加载新的，仍有改善）
- 需要在 `scene.ts` 的渲染循环中增加过渡更新逻辑

**验证**：快速切换两个 VMD，确认过渡期间动作平滑融合而非瞬间跳变

### 8. 程序化动作足部 IK 驱动（procedural-motion.ts）

**问题**：`BONE_LEG_IK_L_CANDIDATES` 和 `BONE_LEG_IK_R_CANDIDATES`（L57-58）已预留但未启用。Auto Dance 时脚完全不动，上半身大幅摆动时下半身像钉在地上。

**修改方案**：

文件：`frontend/src/motion/procedural-motion.ts`

1. 启用足部 IK 骨骼匹配：
```ts
const legIkL = _matchBone(boneNames, BONE_LEG_IK_L_CANDIDATES);
const legIkR = _matchBone(boneNames, BONE_LEG_IK_R_CANDIDATES);
```

2. 在 Auto Dance 的骨骼生成中，添加基础踏步逻辑（在「手首」段之后）：
```ts
// ---------- 11. 足IK（基础踏步） ----------
if (legIkL || legIkR) {
    const stepAmp = 0.08 * intensity;  // Z轴前后位移
    const liftAmp = 0.03 * intensity;  // Y轴抬脚
    const stepPhase = beatFrames / 2;   // 半拍交替
    for (let f = 0; f <= loopFrames; f += 3) {
        const s = sinVals[f];
        // 左脚：正半拍前移，负半拍后移
        if (legIkL) {
            const lz = _clamp1(s * stepAmp);
            const ly = Math.max(0, s) * liftAmp; // 只抬不沉
            bones.push({ name: legIkL, frame: f, position: [0, ly, lz], rotation: [0, 0, 0, 1] });
        }
        // 右脚：反相
        if (legIkR) {
            const rz = _clamp1(-s * stepAmp);
            const ry = Math.max(0, -s) * liftAmp;
            bones.push({ name: legIkR, frame: f, position: [0, ry, rz], rotation: [0, 0, 0, 1] });
        }
    }
    if (legIkL) bones.push({ name: legIkL, frame: loopFrames, position: [0, 0, 0], rotation: [0, 0, 0, 1] });
    if (legIkR) bones.push({ name: legIkR, frame: loopFrames, position: [0, 0, 0], rotation: [0, 0, 0, 1] });
}
```

3. 确保 IK 位移幅度不导致脚穿地面（`liftAmp` 只取正值 = 只抬不沉）。

**潜在影响**：
- 足 IK 骨骼只在模型有对应骨骼时才驱动（`_matchBone` 返回 null 则跳过）
- 位移幅度较小（0.08 * intensity），不会大幅改变整体动作
- MMD 的 IK 链会自动解算膝盖/脚踝，只需驱动 IK 目标点
- 如果模型无 IK 骨骼但有直接腿部骨骼，不会生效（需额外匹配 `左足`/`右足` 候选）

**验证**：加载有 IK 的模型，确认 Auto Dance 时双脚有交替踏步动作
