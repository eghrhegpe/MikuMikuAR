## 二、阶段任务分解

### Phase 1：修复性能模式持久化（P1，~30 行改动）

> 风险：低。Go struct 加字段向后兼容（JSON 反序列化忽略未知字段）。
> 影响：3 文件，改动量极小。

| 1.2 | 新增 `SetPerformanceMode` binding | `library.go`（`SetUIBlurBg` 附近） | +3 行：`func (a *App) SetPerformanceMode(mode string) error { return a.updateConfig(func(cfg *Config) { cfg.UIState.PerformanceMode = mode }, false) }` | `go build` 通过 |
| 1.3 | 前端 import 新 binding | `settings.ts:6-33` | import 列表加 `SetPerformanceMode` | `tsc --noEmit` 通过 |
| 1.4 | `buildSettingsPerformanceLevel` 中 `row.addEventListener` 改为调用 `SetPerformanceMode(m.key)` | `settings.ts:1052-1068` | 替换注释代码为实际调用 | 手动：切模式→重启→确认持久 |
| 1.5 | 确认 `main.ts:473-474` 的恢复逻辑无需改动 | `main.ts` | 无改动（已写好） | — |

#### 影响分析

- **向后兼容**：Go struct 加可选字段，旧 config.json 无此字段时反序列化为零值（空字符串），前端 `if (s.performanceMode)` 跳过，不影响行为
- **Wails binding 自动生成**：`go build` 后 `wailsjs/go/main/App.d.ts` 自动更新
- **无 breaking change**：现有功能不受影响

> 风险：中低。函数迁出 + import 重连，逻辑零改动。
> 影响：2 文件（新增 1 + 修改 1），无 Go 改动。

#### 任务清单

- **减少并发冲突**：settings.ts 从 1118 行降至 ~850 行，多 AI 并发改设置页时冲突概率降低 24%

## 四、执行顺序与依赖

```
Phase 1（P1 持久化）    ← 无依赖，最先做
    ↓
Phase 2（P2 拆分软件）  ← 依赖 Phase 1 完成后 settings.ts 稳定

    ↓

    ↓
Phase 5（P3 恢复默认）  ← 依赖 Phase 1（需要 SetPerformanceMode binding）
    ↓
Phase 6（P3 cs-row 提取）← 可选，依赖 Phase 4（布局重构后再提取）
```

**推荐执行顺序**：P1 → P2+P3 并行 → P4 → P5 → P6（可选）
