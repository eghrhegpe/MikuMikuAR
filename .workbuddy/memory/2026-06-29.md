## Externalize Babylon.js + 性能调优

### 问题
前端构建时间 ~74s（主要瓶颈：Babylon.js 6.8MB + 3171 模块），因此需要获取

### 方案：Externalize @babylonjs/core + @babylonjs/materials

**生产构建**：
- `vite.config.ts` 中 `external` 排除 Babylon 包，`format: 'iife'` 用 `output.globals` 映射到 `BABYLON`（关键：ES 格式下 globals 不生效，必须 IIFE）
- `index.html` 中通过 `<script src="/lib/babylon.js">` 加载 CDN UMD（8.3MB）

**开发构建（wails dev）**：
- 通过 `transformIndexHtml` 插件自动移除 UMD script 标签（避免冗余加载 8.3MB）
- `@babylonjs/core` 保持 `optimizeDeps.exclude`（因 `.fx` shader 文件 ESBuild 不支持）
- Vite dev server 直接 serve node_modules 的 ESM 模块

### 性能问题：按钮点击卡顿 2s

根因：`setEnvState()` → `applyEnvState()` 每次都全量重建环境资源：
- `createClouds()`：`perlin2()` 524,288 次调用，每次重新创建 256 元素排列数组 → ~1.5s
- `createWater()`：每次销毁重建 WaterMaterial → ~200ms
- `createParticleEmitter()`：每次销毁重建 GPU Particle 系统 → ~100ms

**修复**：
1. ✅ `perlin2()` 排列数组提升为模块级常量 `_perlinPerm`，抽取辅助函数 → 524,288 次调用从 ~1.5s 降为 ~20ms
2. ✅ `createClouds()` 已存在则只更新位置/缩放，不重新生成 512x512 噪声纹理
3. ✅ `createWater()` 已存在则直接更新参数（颜色、透明度、波浪等），不重建
4. ✅ `createParticleEmitter()` 已存在则直接跳过重建

### 效果
- 构建时间：~74s → **~1s**（98.7% 提升）
- 模块数：3171 → 127（96% 减少）
- 按钮点击：~2s → **~20-100ms**（跳过重建后）

### 变更文件
- `frontend/vite.config.ts` — 全面重写，Externalize + IIFE + 插件
- `frontend/index.html` — 添加 Babylon UMD script 标签（head）
- `public/lib/babylon.js` — **新增** CDN 下载 8.3MB
- `public/lib/babylonjs.materials.min.js` — **新增** CDN 下载 332KB
- `frontend/src/scene/scene-env-impl.ts` — Perlin 优化 + 重建跳过逻辑

## 五弹窗统一容器改造（Phase 1-3）

### 背景
五个弹窗（模型/动作/设置/场景/环境）分属四个独立 DOM 容器，导致 `toggleOverlay` 行为不一致、关闭逻辑分散、CSS 重复。

### 改造内容

#### 核心变更：所有弹窗统一到 `sceneOverlay`

| 文件 | 修改 |
|------|------|
| `library-core.ts` | `makeModelStack()` container 改为 `dom.sceneOverlay`；`showModelPopup()` 移除自管理生命周期，改为渲染到 `sceneOverlay` |
| `motion-popup.ts` | `makeMotionStack()` container 改为 `dom.sceneOverlay`，`onClose` 改为 `closeAllOverlays`；`showMotionPopup()` 同上 |
| `settings.ts` | `showSettings()` container 改为 `dom.sceneOverlay`，移除自管理生命周期 |
| `scene-menu.ts` | `showSceneMenu()` / `showEnvMenu()` 移除 `closeAllOverlays()` 和 `visible` 自管理，添加 `popupType` 标记 |
| `main.ts` | `navActions` 和 `init()` 的 `addEventListener` 全部改为 `"sceneOverlay"`；`syncNavAriaExpanded()` 改为按 `popupType` 判断活动按钮 |
| `config.ts` | `closeAllOverlays()` 选择器从 `.overlay.visible` 改为 `[data-overlay].visible` |
| `app.css` | 移除 `#modelPopup` / `#motionPopup` 独立 CSS 块，统一到 `#sceneOverlay`，添加宽度 Modifier 类（`.overlay-model` 280px / `.overlay-motion` 320px / `.overlay-settings` 340px）|
| `index.html` | 五个按钮的 `aria-controls` 全部改为 `"sceneOverlay"`，添加 `data-popup-type` 属性 |

#### 弹窗类型标记机制
每个 `showXxx()` 函数设置 `dom.sceneOverlay.dataset.popupType`：
- 模型 → `"model"`
- 动作 → `"motion"`
- 设置 → `"settings"`
- 场景 → `"scene"`
- 环境 → `"env"`

`refreshLibrary()` 据此判断是否需要刷新模型弹窗内容。

## 五弹窗 toggle 修复（最终版，验证通过）

### 问题现象
关闭弹窗后无法重新打开；`aria-expanded` 正常变换，但弹窗 DOM 不显示。
用户诊断：「感觉是时序、多个代码同时影响导致的」。

### 根因（最终确认）
`showXxx()` 函数在 `innerHTML = ""` 清空 `sceneOverlay` 后，复用了旧的 `SlideMenu` 实例。
该实例持有已被销毁的 DOM 节点引用，`reset()` 调用时操作的是**已分离的 DOM**，
内容实际未渲染到 `sceneOverlay`——看起来「打不开」，实则是内容从未被挂载。

### 修复方案
五个弹窗的 `showXxx()` 函数，全部改为**每次 `new SlideMenu()`**，不再复用旧实例。

| 文件 | 函数 | 修复 |
|------|------|------|
| `library-core.ts` | `showModelPopup()` | ✅ 每次重建 `modelStack` |
| `motion-popup.ts` | `showMotionPopup()` | ✅ 每次重建 `motionStack` |
| `settings.ts` | `showSettings()` | ✅ 每次重建 `settingsStack` |
| `scene-menu.ts` | `showSceneMenu()` | ✅ 每次重建 `sceneStack` |
| `env-menu.ts` | `showEnvMenu()` | ✅ 每次重建 `envStack` |

### 同步修复（本次会话）
| # | 问题 | 修复 |
|---|------|------|
| 1 | `overlay-fade-out` 残留导致弹窗透明 | `closeAllOverlays()` 添加 `classList.remove("overlay-fade-out")` |
| 2 | `toggleOverlay` 打开分支未清理残留类 | 添加 `el.classList.remove("overlay-fade-out")` |
| 3 | `syncNavAriaExpanded()` 位置错误（嵌套在 `if` 块内） | 移到 `if` 块外，确保总是执行 |
| 4 | Canvas 点击恢复弹窗导致状态不同步 | 废除 `_overlaysHiddenByClick` / `_lastHidden`，Canvas 点击只关闭不恢复 |
