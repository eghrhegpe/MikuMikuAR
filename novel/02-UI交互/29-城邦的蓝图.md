# 城邦的蓝图

> **背景**：menus/ 目录膨胀到 16,345 行 / 41 个文件，全部为命令式 builder，AI 难以重写。ADR-093 决策引入声明式 Schema + 单一渲染器。
> **过程**：定义 MenuNode 类型 → StatePath 前缀体系 → menu-schema.ts 数据源 → menu-factory.ts 统一渲染 → 57 个面板迁移。

---

## 序、两张图

王逸打开 `scene-menu.ts`，滚动到第 200 行。

`SCENE_FOLDER_ROUTES` 是一个 `Record<string, () => PopupLevel>`——静态路由表，点菜单文件夹时查这张表，找到对应的 build 函数，执行它，返回那一层。

这是联邦的导航机制之一。

但他又打开 `scene-prop-levels.ts`，在第 33 行看到了另一样东西：`getXMenu()?.push(buildX(id))`。

这也是导航。但它是动态的——点了某个道具的详情，`push` 一个新的层级进栈。它不走路由表，它自己建。

两套机制。两套代码路径。静态的走 `Record`，动态的走 `push`。

王逸盯着这两行代码，想到了一件事：如果城邦的街道分布只有一张图，那 AI 就能读懂。如果一张在纸上，一张在脑子里，那就没人能读懂。

---

## 一、一张图

ADR-093 的核心只有一句话：**菜单树应该是一份数据，不是一堆函数。**

`MenuNode` 是这个数据的载体。每个节点有一个 `id`——稳定的、唯一的。`kind` 告诉渲染器它是什么：文件夹、滑块、开关、动态列表。`label` 是一个 i18n key。`children` 是它的子节点。

```typescript
interface MenuNode {
    id: string;
    kind: MenuKind;
    label?: string;
    children?: MenuNode[];
    control?: ControlSpec;
}
```

`id` 是路由的基础。`ground:basic` 指向地面基础设置层。`env:cloud` 指向云层设置层。不再需要魔法字符串 `scene:render:dof`——拼写错了编译器不会提示，但 `id` 是类型安全的。

命令式 builder 没有消失。它们降级为"自定义层渲染后端"。表单类特殊层仍然可以用命令式写法，但标准层——滑块、开关、文件夹——都从 schema 派生。

---

## 二、门牌号

StatePath 是 schema 的第二个关键设计。

```typescript
type StatePath = `env.${string}` | `render.${string}` | `light.${string}` | `ui.${string}` | `perception.${string}`;
```

每个控件绑定一个 StatePath。解析器按前缀映射到 reactive state 对象。`env.groundVisible` 映射到 `envState.groundVisible`。`render.dofStrength` 映射到 `getRenderState().dofStrength`。

以前，每个滑块的 `onChange` 闭包里都写着 `setEnvState({ groundVisible: v })`。改一个控件要全网搜它的 `onChange`。现在，控件绑定 StatePath，状态变化自动同步。

就像给每个房间装了门牌号。你不需要进屋才知道这是谁的房间。

---

## 三、工厂

`menu-schema.ts` 定义数据。`menu-factory.ts` 负责渲染。

工厂接收一个 `MenuNode` 树，遍历每个节点，根据 `kind` 创建对应的 DOM 元素。读取 `control.bind`，从 reactive state 读取当前值。绑定 `onChange`，写回 reactive state。

写完后，王逸打开应用，打开场景菜单。

菜单结构和以前一模一样。同样的层级、同样的滑块、同样的开关。

但底层变了。41 个函数变成了一份数据。两套导航机制变成了一套。魔法字符串变成了类型安全的 StatePath。

---

## 四、迁移

最难的不是写新代码，是把旧的搬过来。

57 个面板。每个面板都有一堆 `build*Level()` 函数。王逸从最简单的开始——地面面板的基础设置。

旧代码：

```typescript
slideRow({
    label: '地面可见度',
    icon: 'mdi:grid',
    get: () => envState.groundVisible,
    set: (v) => setEnvState({ groundVisible: v }),
});
```

新代码：

```typescript
{
    id: 'ground:visible',
    kind: 'slider',
    label: 'menu.ground.visible',
    icon: 'mdi:grid',
    control: { bind: 'env.groundVisible', min: 0, max: 1, step: 0.01 },
}
```

少了一半。旧代码里有闭包、有副作用、有硬编码字符串。新代码里只有数据。

动态节点更难。模型列表的子项数量在运行时才确定。旧代码里是 `buildModelListLevel()` 函数，运行时调用 `modelManager.getModels()` 生成列表。新代码里，`kind` 设为 `dynamic`，`childrenResolver` 接收数据数组，返回子节点数组。

静态 `children` 和动态 `childrenResolver` 都从同一份 schema 派生。不再有 `push` 旁路。

## 尾声

57 个面板迁移完的那一天，王逸打开设置页，又打开场景页，又打开动作页。

每个菜单都是一份数据渲染出来的。同样的层级，同样的控件，同样的状态绑定。但底层不再是 41 个函数的集合，而是一张图。

城邦的蓝图，终于只有一张了。
