# 五种语言

> **背景**：UI 文本硬编码中文，无法切换语言。i18n 系统需要支持五种语言。
> **过程**：locale.ts + t.ts → 五语言翻译文件 → 菜单/提示/状态栏全部走 t() → 语言切换实时生效。

---

## 一、硬编码的墙

联邦的界面从一开始就是中文的。

`"地面可见度"`、`"模型库"`、`"场景设置"`——这些字符串直接写在代码里，嵌在 `slideRow` 和 `buildXxxLevel` 的调用中。它们不是数据，它们是代码的一部分。

Sisyphus 知道这是个问题。但他一直觉得"先让东西能跑"。

直到有一天，用户在 GitHub 上提了一个 issue："Can this support English?"

问题不在翻译。问题在于——字符串不在一个地方。它们散在三十多个菜单文件里，每改一个地方都要改代码。

Sisyphus 决定建一个翻译层。

---

## 二、t 函数

翻译层的入口是一个叫 `t` 的函数。

```typescript
function t(key: string, params?: Record<string, any>): string
```

它接收一个 key，比如 `"menu.ground.visible"`。然后在当前语言的翻译表里查找对应的文本。找不到就返回 key 本身——这是一种防御：翻译缺失时，至少能告诉用户这个 UI 元素叫什麼。

如果传了 params，就替换 `{name}` 这样的占位符。

```typescript
t('model.loaded', { name: '初音未来' })
// → "模型 初音未来 已加载"
```

翻译表是五个文件：`zh-CN.ts`、`en.ts`、`ja.ts`、`ko.ts`、`zh-TW.ts`。每个文件是一个扁平的 `Record<string, string>`。

```typescript
export const zhCN: Record<string, string> = {
    'menu.ground.visible': '地面可见度',
    'menu.ground.size': '地面大小',
    // ...
};
```

五种语言，五个文件。每种语言的翻译量差不多——五百多个 key，每种语言五百多条文本。

---

## 三、替换

最难的不是写翻译，是把现有的硬编码字符串替换成 `t()` 调用。

Sisyphus 写了一个脚本，扫描所有 `slideRow`、`addSliderRow`、`addToggleRow` 的 `label` 参数，提取中文字符串，生成 i18n key。

但脚本只能做一半。有些字符串是动态拼接的，有些是放在 `toast` 里的，有些是放在 `dialog` 的标题里的。这些要手动改。

手动改了三千多处。

改完之后，代码里不再出现 `"地面可见度"`。取而代之的是 `t('menu.ground.visible')`。

---

## 四、热切换

语言切换需要实时生效。

用户点击"English"，所有 UI 文本立刻变成英文。不需要重启。

Sisyphus 在 `setLocale` 里加了一个刷新机制：

```typescript
function setLocale(locale: Locale): void {
    currentLocale = locale;
    localStorage.setItem('locale', locale);
    
    const currentMenu = getActivePopupMenu();
    if (currentMenu) {
        currentMenu.refreshRoot();
    }
}
```

`refreshRoot` 会重新执行 `buildRootItems`。因为所有 label 都是 `t(key)`，语言变了，翻译就变了。菜单自动更新。

---

## 尾声

五种语言。不是翻译，是联邦的边界扩展。

字符串从代码里移出来了，放在翻译表里。改一个翻译，不需要改代码，不需要重新构建。

联邦不再只有一个声音。
