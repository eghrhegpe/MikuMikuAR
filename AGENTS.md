## 本地缓存（多 AI 协同规范）

> 禁止使用 `git stash`，会把文件挪走，多进程并发 pop 时会产生大量误解。
> 改完记得本地缓存：git add . && git commit 。
> 推送/PR：建议把所有改动都提交，按 Safety Protocol 用具体文件名，或至少 git add -A 前先 git status 确认范围

```bash
# 暂存（本地缓存）
git add .
git commit -m "<type>: <简短描述>"    # 不带 --no-verify，避免触发 pre-commit

# 恢复（从本地缓存取出）
git reset --soft HEAD~1               # 撤销最近一条 commit，把改动放回暂存区
# 或
git reset HEAD~1                       # 撤销最近一条 commit，把改动放回工作区
```

| 规则 | 说明 |
|------|------|
| commit 信息格式 | `<type>: <描述>`，type 同conventional commits（feat/fix/docs/chore/refactor/test） |
| 推送前 squash | 多个本地 commit 可以 `git rebase -i` 合并为一个有意义的 PR commit |
| 禁止 | `git stash` / `git stash push` / `git stash pop` |

# 审核代码可用性

> 按功能模块依次遍历，每模块需验证 5 个维度：类型安全、资源管理、测试覆盖、功能正确性、设计质量。
>项目已有 docs/audit/ 目录，可参考。
> 发现预料之外的缺陷时，只报告，给出精确的修复建议（diff 格式、文件:行号、修改原因），

## 代码审核维度标准

| 维度 | 检查项 | 通过标准 |
|------|--------|---------|
| **类型安全** | `as any` / `@ts-ignore` | 生产代码中 0 处新增；遗留需有业务理由注释 |
| **资源管理** | `.dispose()` 配对 | 每个 `new Xxx()` Babylon / WebAudio 对象有对应释放；observer 在 dispose 时移除 |
| **测试覆盖** | 直接测试 / 间接覆盖 | 核心逻辑有单元测试；UI builder 允许无测试 |
| **功能正确性** | 运行时隐患 | 并发守护 / undefined 守卫 / Promise 不丢弃 / 竞态处理 |
| **设计质量** | 架构、模式、可维护性 | 职责边界清晰；状态管理可追踪；无隐式耦合 |
| **设计质量2** | 前端易用性 | 样式系统（CSS 变量复用、布局一致性）与 UI 代码组织结构（功能分类清晰、组件拆分合理） |

## 审核思维准则（审查员视角）

> AI 执行审核时，不是逐行读代码，而是带着以下 4 个问题去"盘问"代码：

| 思维模型 | 核心问题 | 审核时怎么做 |
|----------|---------|-------------|
| **数据流追踪** | 这个状态从哪来？经过谁修改？最终流到哪？ | 追踪 `envState.xxx` 的所有写入点（`grep` setter / `setEnvState`），确认无"幽灵路径"（模块级变量被多函数隐式修改） |
| **生命周期完整性** | 对象创建和销毁是否在同一抽象层次配对？ | `initLighting` 里 `new` 的东西，必须在 `disposeRenderer` 或同级 dispose 中释放，不能在随意角落处理 |
| **并发与边界** | 异步操作是否有过期标记？快速触发是否竞态？ | 检查 `_loading` / `_pending` / generation counter 是否存在；模拟用户连点 3 次的场景 |
| **异常契约** | 函数抛出异常后，调用方是否还能安全使用该模块？ | 检查 `catch` 后是否将对象置为一致状态；`finally` 中是否执行了必要的清理；Promise 是否将异常转化为可预期的返回值 |

**心理模拟步骤：**
1. **契约检查**：该模块的公开函数签名是否与内部实现一致？是否有隐式依赖外部全局状态？
2. **状态机模拟**：如果用户快速点击 3 次不同功能，模块内的 `_loading` / `_pending` 标志是否能正确拦截？
3. **异常模拟**：如果第 50 行抛出异常，第 100 行的 `dispose` 是否还会执行？（`finally` 是否覆盖）
4. **引用计数检查**：`addEventListener` 是否都有对应的 `removeEventListener`？`Observable.add` 是否在 `dispose` 中被移除？

## 设计质量检查项

| 检查项 | 说明 | 示例（好） | 示例（差） |
|--------|------|-----------|-----------|
| **状态来源唯一** | 同一状态不应从多处读写 | `envState` 是唯一来源 | `store/index.ts` + `state.ts` 双源 |
| **副作用可追踪** | 函数不应隐式修改外部状态 | 通过参数显式传递 | 模块级变量被多处直接写入 |
| **并发安全** | 异步操作有去重/锁 | `_loadingOutfits` Set 去重 | 多次点击触发重复加载 |
| **错误边界** | 异常不吞没、不扩散 | `try/catch` + `console.warn` + 用户提示 | 静默 `catch {}` 或 Promise 无 `.catch` ，所有耗时 `async` 调用是否接受 `AbortSignal`？`finally` 块中是否对 `disposed` 标志进行守卫？ |
| **资源释放链路** | dispose 应级联释放子资源 | `disposeWater()` 释放 RT + camera + observer | 只释放顶层，子资源泄漏 |
| **i18n 国际化完整性** | 搜索 `innerText` / `textContent` / `innerHTML` 直接赋值，确认是否使用 `t()` 包裹 |

## 设计质量 — 常见反模式（审查时重点排查）

| 反模式 | 表现 | 危害 |
|--------|------|------|
| **隐式状态写入** | 函数直接修改 `_xxx` 全局变量，而非通过 setter/action | 状态变化不可追踪，难以 debug |
| **循环依赖假象** | A 导入 B，B 导入 A（运行时可通过延迟加载绕开，但耦合已存在） | 增加重构成本 |
| **职责过载** | 一个函数做了"数据获取 + UI 更新 + 状态持久化" | 违反单一职责，难以测试 |
| **魔法数值/硬编码** | `if (x > 0.5)` 或 `'scene:save'` 无常量定义 | 修改时极易遗漏 |
| **显著重复** | 相似逻辑在 **≥2 个文件**中出现（UI builder 纯布局重复除外） | 应抽取为 `ui-helpers` 或公共函数；若仅 2 处且语义独立，需在 ADR 中说明豁免理由 |
| **Promise 链断裂** | async 函数中 `.then()` 无 `.catch()`，或 `catch` 后静默吞错 | 错误无声消失，用户不知发生了什么 |

# 交互可用性（UX）审核 — 通过代码模式识别体验问题

> **核心原则**：代码在类型上正确、资源上不泄漏，不等于用户在界面上「好用」。审核时需从代码中提取交互路径，模拟用户的操作序列，识别以下 5 类体验隐患。

| UX 维度 | 检查方式（阅读代码） | 好模式 | 差模式（警示） |
|---------|---------------------|--------|---------------|
| **操作路径深度** | 统计从根菜单到目标功能的 `push` / `show` 层级数 | 核心功能 ≤3 层可达 | 设置/灯光等高频功能藏于 ≥5 层嵌套（`folder` 套 `folder` 套 `folder`） |
| **操作反馈（异步）** | 查找 `async` 操作前后是否更新 UI 状态 | `setStatus('加载中...')` → `await` → `setStatus('完成')` | 点击按钮后无任何视觉反馈，用户重复点击触发竞态 |
| **破坏性操作防呆** | 查找 `remove` / `delete` / `reset` 是否前置确认 | `showConfirm('确定删除？')` 后才执行 | 直接执行删除，无二次确认或撤销途径 |
| **错误消息可理解性** | 查找 `catch` 块中抛给用户的文本 | `t('toast.fileNotFound', { name })` 含具体文件名 | `catch(e) { alert(e.message) }` 抛技术栈给用户 |
| **交互一致性** | 同类操作是否复用相同 UI 组件 | 所有滑块用 `addSliderRow`，所有开关用 `addToggleRow` | 灯光用自定义滑块，材质用另一套滑块，行为/样式不一致 |
| **空状态与引导** | 首次打开无数据时，界面是否提供行动入口 | 空列表显示「暂无道具，点击加载」+ 按钮 | 空白界面无任何提示，用户不知道下一步该做什么 |
| **操作结果可撤销** | 破坏性操作（重置、批量删除）后是否有撤销或「恢复」路径 | 重置后有「撤销」按钮，或状态可回滚 | 操作一次性完成，无任何回退机制 |

**UI Builder 特有检查项：**
| 检查点 | 好 | 差 |
|--------|----|----|
| 滑块数值范围 | 取值范围有明确语义（如曝光 0–4，步长 0.05） | 取值范围 0–100 但无单位，用户不知调的是什么 |
| 开关状态同步 | 开关与 `bind()` 绑定，状态变化时 UI 同步更新 | 开关状态只写一次，外部状态变化后 UI 显示过时 |
| 列表项操作 | 每个列表项有「详情」入口 + 「删除」独立按钮 | 点击整行即删除（误触风险），或长按才弹出菜单（发现性差） |
| 加载状态占位 | 异步加载时显示骨架屏 / 旋转器 / 灰色占位 | 列表突然从有变无，内容闪烁或跳变 |
