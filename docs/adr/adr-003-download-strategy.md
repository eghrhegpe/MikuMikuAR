# 下载策略决策 — ADR-003

**日期**：2026-07-16

---

### 背景

模之屋下载接管原计划走 WebView 内嵌 + 下载拦截，但经过技术评估发现三个风险：
1. Wails v2 未暴露 WebView2 的 DownloadStarting 事件，需用 `<iframe>` + JS hook 替代方案
2. 模之屋可能设 `X-Frame-Options: DENY`、下载需登录、下载 URL 临时签名 → 每个都是未知阻塞
3. 模之屋改版需持续维护注入 JS，长期成本高

### 替代方案对比

| 方案 | 体验 | 复杂度 | 维护成本 | 反爬风险 |
|------|------|--------|---------|---------|
| A. WebView 内嵌 + 拦截 | 最好 | 高 | 高 | 高 |
| B. 监听下载目录 + 自动导入 | 中 | 低 | 低 | 无 |
| C. 维持现状 | 差 | 零 | 零 | 无 |

### 决定

选 **方案 B**：

1. 用 `fsnotify` 监听系统下载目录（`%USERPROFILE%/Downloads/`），新 `.zip/.rar/.7z` 文件出现时弹通知
2. 用户在浏览器的任何站点下载模型 → 自动检测 → 一键导入（复现 `ExtractZip` 链路）
3. 不绑定模之屋单点，对所有下载来源通用

### 理由

- 覆盖了「自动落库」核心价值，用户不用手动拖文件
- 零反爬风险，不依赖任何站点的 DOM 结构或 API
- 实现成本低：1 个 Go binding + `fsnotify` 监听

### 排除

- 放弃 WebView 内嵌方案（除非未来 Wails 原生暴露 DownloadStarting 事件）
- 不监听除下载目录外的位置（降低误触风险）
