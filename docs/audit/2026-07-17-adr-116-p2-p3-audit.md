# ADR-116 P2+P3 动作系统审核报告

> 审核日期: 2026-07-17
> 审核范围: P2+P3 交付物（commit `a3afc9d`，17 文件 +1042/-23）
> 审核对象: `bone-override.ts`(setBoneOverridePosition 双模式) / `body-posture.ts`(height/fwdBack 位置覆盖) / `scene-serialize.ts`(持久化) / `head-tracking.ts`(新,perception 通道) / `sway-motion.ts`(新) / `finger-pose.ts`(新) / `riding-model.ts`(新) / `registry.ts`(4 注册 + 仲裁)
> 审核方法: AGENTS.md 五步法（导入图谱 → 状态读写追踪 → 资源配对 → 心理模拟 → 报告）
> 边界说明: P0+P1 已于 `2026-07-17-adr-116-p0-p1-audit.md` 审核，其 P1 缺陷在 `dfcc4b5` 修复；本报告**不复审** P0+P1 已覆盖的 body-posture/hand-symmetry/registry 基础机制，仅聚焦 P2+P3 增量及其与既有系统的交互。

## 总体结论

**有条件通过（可发布，含 1 项 P2 功能缺陷 + 若干 P3/P4）。**

- 架构契约正确：`setParam` 启用门控 + 拖滑块自动 `enable`（VRChat 行为）、`disable` 精确清 `ownedBones` 不误伤手动覆盖、i18n 5 locale 全 key 就绪、类型安全零 `as any`/`@ts-ignore`、持久化 `setState+enable` 重烘焙 + defaults 兜底，均达标。
- **核心 P2 缺陷**：`claimBones` 冲突仲裁采用**先到先得**，**完全不参与 `priority` 数值比较**。而 `body-posture`(p1) 与 `sway-motion`(p3)/`riding-model`(p3) 共享 `センター`/`腰` 骨骼——这正是 P0+P1 审计中预警「priority=1 且骨不重叠暂无实害，P2/P3 埋雷」的那颗雷，现已引爆：用户若先启用 sway/riding 再启用 body-posture，body-posture 的位置覆盖被静默拒绝（仅 `console.warn`），开关显示 ON 却无 3D 效果。
- P3 主要为体验/健壮性：sway 的 frequency/decay 滑块无 3D 效果、head-tracking per-model 模块操作全局开关（依赖 `setTargetModel` 批量同步才一致）、4 个新模块无模块级单测。

## F1 推荐修复 diff（P2 必须项 — 仲裁引入 priority 抢占）

```ts
// registry.ts —— claimBones 引入 priority 比较 + 抢占清 slot
export function claimBones(modelId: string, moduleId: string, bones: readonly string[]): string[] {
    const owned = _ownedMap(modelId);
    let mySet = owned.get(moduleId);
    if (!mySet) {
        mySet = new Set();
        owned.set(moduleId, mySet);
    }
    const myPriority = _registry.get(moduleId)?.priority ?? Number.MAX_SAFE_INTEGER;
    const claimed: string[] = [];
    for (const bone of bones) {
        if (mySet.has(bone)) {
            claimed.push(bone);
            continue;
        }
        // 找当前占用者
        let conflictOwner: string | null = null;
        for (const [otherId, otherSet] of owned) {
            if (otherId !== moduleId && otherSet.has(bone)) {
                conflictOwner = otherId;
                break;
            }
        }
        if (conflictOwner) {
            const otherPriority = _registry.get(conflictOwner)?.priority ?? Number.MAX_SAFE_INTEGER;
            if (myPriority < otherPriority) {
                // 高优先级抢占：释放冲突方该骨登记 + 清其引擎 slot（消除落败方残留覆盖）
                owned.get(conflictOwner)?.delete(bone);
                clearBoneOverride(bone, modelId);
                console.warn(
                    `[adr-116] 模块 "${moduleId}"(p${myPriority}) 抢占骨骼 "${bone}"，释放模块 "${conflictOwner}"(p${otherPriority})`
                );
            } else {
                console.warn(
                    `[adr-116] bone "${bone}" 已被模块 "${conflictOwner}"(p${otherPriority}) 占用，模块 "${moduleId}"(p${myPriority}) 跳过`
                );
                continue;
            }
        }
        mySet.add(bone);
        claimed.push(bone);
    }
    return claimed;
}
```

> **配套影响**：现有单测 `claimBones 冲突：已被其他模块占用的骨骼被跳过并 warn`（body-posture p1 vs hand-symmetry）仍通过——body-posture 先声明 `上半身` 且无冲突，hand-symmetry 后声明遇冲突且优先级更高数值（被拒），期望 `claimed=['左腕']` 不变。F1 仅改变"高优先级晚到可反抢"的行为，对此用例无影响。
> **落败方状态**：被抢占模块仍显示 enabled=ON，但其骨贡献被抑制——符合"高优先级模块赢了这根骨"的语义；其 `owned` 登记已移除，后续 `disable` 不会误清（也不会重复清）。若被抢占方需"让位提示"，可在 UI 层监听该 warn 文案做轻量提示（非必须）。

## 验收建议

1. **P2 修复（F1）后必做**：`cd frontend && npm run build && npm run test`（registry 单测 + 全量 1557）。
