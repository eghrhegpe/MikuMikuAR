🔴 P0 问题（功能性缺陷）
1. reattachPipeline 未在 switchCameraMode 中调用 — 偏离计划

实施计划第 24、49、170-171 行明确要求：在 camera.ts 的 switchCameraMode 末尾调用 reattachPipeline()，并把当前 FOV 应用到新相机。但实际代码中 camera.ts:117-181 的 switchCameraMode 函数末尾什么都没加。

取而代之，reattachPipeline() 被放在了 scene-menu.ts:365 的 handleSceneAction 里：

scene-menu.tsL360-L370
    if (row.target && row.target.startsWith("camera:")) {
        const mode = row.target.replace("camera:", "") as "orbit" | "freefly" | "oneshot" | "concert";
        switchCameraMode(mode);
        reattachPipeline();  // re-attach post-processing pipeline to the new camera
问题：如果 switchCameraMode 从菜单以外的路径被调用（快捷键、其他模块），后处理管线不会重挂，Bloom/FXAA 会失效。计划中提到的风险点"switchCameraMode 后 pipeline.cameras 未更新 = 后处理失效"实际上并未在 camera.ts 层面修复。

2. 新相机的 FOV 未应用

计划第 48-50 行要求：切换相机后，把用户配置的 fov 应用到新相机。camera.ts:174 只设了 scene.activeCamera = newCam，没有设 newCam.fov。用户在舞台面板调过 FOV 后，切相机会回到默认 FOV。

3. getRenderState() 永远返回 outlineEnabled: false

scene.tsL131-L147
export function getRenderState(): RenderState {
    const cam = scene.activeCamera;
    return {
        bloomEnabled: pipeline.bloomEnabled,
        bloomWeight: pipeline.bloomWeight ?? 0.3,
        bloomThreshold: pipeline.bloomThreshold ?? 0.5,
        bloomKernel: pipeline.bloomKernel ?? 64,
        outlineEnabled: false, // managed via mesh edges, not pipeline
        outlineColor: [0, 0, 0],
用户打开"边缘高亮"后，关闭面板再打开，开关会显示为关闭状态，但 mesh 上的 edgesRendering 仍然在跑。outlineColor 同理永远返回 [0,0,0]。应该用一个模块级变量追踪 outline 状态。

🟡 P1 问题（逻辑/健壮性）
4. reattachPipeline 用 addCamera 而非替换

计划写的是 pipeline.cameras = [scene.activeCamera]（替换），实际用的是 pipeline.addCamera()（追加）：

scene.tsL196-L201
export function reattachPipeline(): void {
    if (scene.activeCamera) {
        pipeline.addCamera(scene.activeCamera);
    }
}
多次切相机会在 pipeline.cameras 数组里累积已销毁的旧相机引用。虽然旧相机已从 scene 移除，但管线内部仍持有引用，可能导致内存泄漏或渲染异常。应改为 pipeline.cameras = [scene.activeCamera]。

5. 用户预设保存失败时状态不一致

showPresetSaveDialog() 中，SaveRenderPreset 的 .catch() 只 console.warn，但内存中的 userPresets[trimmed] = state 已经先执行了：

scene-menu.tsL330-L337
    const state = getRenderState();
    userPresets[trimmed] = state;
    // Persist via Go backend
    SaveRenderPreset(trimmed, JSON.stringify(state)).catch((err: any) => {
        console.warn("SaveRenderPreset failed:", err);
    });
如果 Go 端写盘失败，内存里有预设但重启后消失，用户无感知。应在 catch 中回滚 delete userPresets[trimmed] 并 setStatus 告知用户。

6. outlineColor 设置逻辑冗余

setRenderState 中有两段遍历 mesh 设 edgesColor 的代码（158-171 和 172-180），当 outlineEnabled 和 outlineColor 同时传入时，颜色会被设两次。不影响正确性，但可合并。

🟢 P2 问题（代码质量）
7. getPresetName 和 buildPresetsLevel 中的 labels 重复定义

scene-menu.ts:299-301 和 318-323 各定义了一份相同的中文标签映射，应提取为模块级常量。

8. loadUserPresets 每次打开菜单都调 Go

showSceneMenu()（line 461）每次都 await loadUserPresets()，即使用户没改过预设也会触发一次 Go 调用。可加脏标记优化。

9. colorCorrectionEnabled 字段缺失（计划偏离）

计划第 37 行的 RenderState 接口含 colorCorrectionEnabled: boolean，实际实现中没有。计划 1.5 节已说明"本次只暴露 contrast + toneMapping"，属有意偏离，可接受。但建议在计划文档中标注。
