# 第 3 章 · 议会与织工

> 对应真实事件：2026-06-30 第三波审计，覆盖 menus/（SlideMenu 生命周期、library-core 加载锁、motion-popup 栈逻辑、model-material 空指针、model-preset XSS 防护、outfit-ui 异步刷新、settings dispose）、motion/（procedural-motion speed 钳位、vmd-writer Shift-JIS + trailer、vpd-parser 正则加固、lipsync 灵敏度边界、beat-detector GainNode）、outfit/（audio.ts seeked 事件 + 加载版本号）、AGENTS.md 文档宪法更新。

---

第二波审计结束时已是深夜。外交官合上笔记本，在物理引擎的子步循环里发现的那个设计缺陷——重力只在全帧上施加一次——像一块冰在他脑子里融化，化成了一摊清醒的凉水。

他不困。他翻开了笔记本的另一页。

今天其实是从菜单开始，围绕菜单结束的。中间是外交官最熟悉也最疏远的东西——织工的工作。

---

## 议会的裂缝

**议会**（`SlideMenu`，`menu.ts`）是联邦所有弹窗的导航系统。第一波审计时它被看了一眼——"动画挺流畅的，没问题"——然后就被跳过了。

外交官现在重新打开它，因为他在光照之城、加载之城、八座城邦的第二波审计中反复看到同一个模式：**每次调用 `showXxxMenu()` 都创建一个新的 `SlideMenu`，从不销毁旧的。**

每次新实例都会注册一个 `keydown` 监听器到容器上。十次打开环境弹窗，就有十个监听器。键盘方向键按下时，十个回调同时触发——九个是僵尸，一个是活着的。僵尸们试图操作已经不在 DOM 中的元素，什么都做不了，但它们的闭包里持有整个 `SlideMenu` 实例的引用——无法被 GC。

还有那些 `setTimeout`。`push` 和 `pop` 的动画中，每个过渡都有后备定时器（150ms、200ms），确保即使 `transitionend` 没触发，过渡状态也能恢复。但如果动画被强制打断——比如 `reset` 或 `popTo`——旧定时器不会被清除。它们会在未来的某个时刻触发，试图把 `transitioning` 置为 `false`，而此时菜单可能正在另一个动画中。

外交官在 `SlideMenu` 内部建了一个定时器追踪数组 `_pendingTimeouts`，在 `_cancelAnim` 中全部清除。他加了 `dispose()` 公有方法——清除定时器、移除 `keydown` 监听器、清空缓存按钮。然后在六个调用点（`showSceneMenu`、`showModelPopup`、`showEnvMenu`、`showMotionPopup`、`showSettings`）加上了一行：`oldMenu?.dispose()`。

---

然后他发现了第二个问题——不是 bug，是**设计缺陷**。

`SlideMenu.buildPanel` 调用 `level.renderCustom(container)`。`renderCustom` 在类型定义中是 `(container: HTMLElement) => void`。但在六个地方——预设场景列表、标签页、舞蹈套装概览——它是 `async (container) => { await ... }`。

TypeScript 没有报错。`Promise<void>` 可以赋值给 `void`。但 `SlideMenu` 不会 `await` 这个 Promise——它在 `renderCustom` 返回后立即设置焦点、触发 `onAfterRender`。异步内容还没渲染完，焦点已经设置好了。在慢速后端调用中，用户会看到短暂闪烁。

外交官把 `PopupLevel.renderCustom` 的类型改为 `void | Promise<void>`，把 `buildPanel` 改为 `async`，在 `push`/`pop` 的过渡回调中加了 `await`。`reset`/`popTo`/`reRender` 用 `.then()` 适配。

从此，异步菜单内容在完成前不会触发焦点和回调。

## 学舌的创伤

第三波审计中，外交官发现了一个全联邦蔓延的模式，他后来称之为"学舌伤"——同一个问题，在不同模块中以几乎相同的方式出现。

**加载锁缺失**。`library-core.ts` 的 `onModelRowClick` 中，zip 解压没有互斥保护。用户快速双击两个模型，两个 `ExtractZip` 同时进行，两个 `loadPMXFile` 排队——都在操作同一组全局状态。外交官加了 `_isExtracting` 锁。

**XSS 风险**。`model-preset.ts` 的 `showUndoToast` 中，预设名称直接拼接进 HTML 字符串。若预设名包含 `<script>`，它就是真的 script。外交官用 `escapeHtml()` 包裹了它。`motion-popup.ts` 中 `setAttribute('data-hint', ...)` 也有同样的问题——改用 `dataset.hint`，浏览器自动转义。

**栈逻辑混乱**。`motion-popup.ts` 的舞蹈套装删除操作中，`pop()` 后跟着 `setLevel(rootIdx, overview)`——如果 pop 后已经在概览层，`setLevel` 又覆盖了一次。外交官改成简单的 `pop()` + `reRender()`。

**空指针崩溃**。`model-material.ts` 的 `_selectedMat.cat` 在首次进入面板时必然为 `null`——这是一个初次访问就必触发的崩溃。外交官把所有 `_selectedMat.cat` 改成了 `_selectedMat?.cat`，并在 `buildMatRootLevel` 开头加了一行 `_selectedMat = null`。

**每帧重绘**。`model-material.ts` 的四个材质滑块，每次拖动都调用 `stackRegistry.modelStack.reRender()`——拖动 100 次 = 100 次全面板重建。外交官移除了滑块回调中的 `reRender()`，保留了点击时的重绘。

**`mmku:modelLoaded` 事件滥用**。`library-core.ts` 监听了这个事件，无论弹窗状态如何，无条件调用 `showModelPopup()`。如果用户正在环境弹窗里调粒子参数，模型加载事件会直接覆盖环境弹窗为模型弹窗。外交官加了守卫：`if (!visible || popupType !== 'model') return`。

学舌伤——同一个 bug 在学舌中传播，从一个模块复制到另一个——是联邦最深的伤口。

## 声音的边界

**音频系统**（`audio.ts`）的问题小而精。

`seekAudio` 用 `setTimeout(50ms)` 重置 `isSeeking` 标志。但 `audioElement.currentTime` 赋值是异步的——50ms 在慢速设备上可能不够，在快速设备上又太长。外交官改用 `seeked` 事件。

`loadAudioFile` 是一个异步函数。用户快速加载两首音乐时，第一个 `loadAudioFile` 正在 `await resolveFileUrl`，第二个已经在路上——两个加载交错执行，写入同一组状态变量。外交官加了一个 `_loadId` 事务 ID：每次加载递增，在 `resolveFileUrl` 和 `play()` 后校验，过期操作自动放弃。

**节拍检测器**（`beat-detector.ts`）的 `attach` 通过 `createMediaElementSource` 将音频元素接入 `AudioContext`。一旦接入，`audioElement.volume` 被旁路——音量滑块形同虚设。外交官在信号链中插入了 `GainNode`，路径变成 `source → analyser → gain → destination`，暴露 `setVolume(value)`。`audio.ts` 的 `setVolume` 同步调用 `beatDetector.setVolume()`。

`new AudioCtx()` 可能因浏览器策略抛出异常。他加了 try-catch。

## 程序的骨骼

**程序化动作生成器**（`procedural-motion.ts`）和 **VMD 写入器**（`vmd-writer.ts`）被一起审计，因为它们是共生关系——一个生成数据，一个写入文件。

`procedural-motion.ts` 的 `generateIdleVmd` 中，循环帧数 = `Math.round(120 / speed)`。如果外部传入 `speed=0`，结果是 `Infinity`——后续的 `for (let f = 0; f <= Infinity; f += 6)` 会导致内存瞬间耗尽。外交官加了三级防护：safeSpeed 钳位到 0.1~10，帧数上限 600（20 秒），眨眼帧循环条件 `t + 5 <= loopFrames`。

四元数归一化计算 `w = sqrt(1 - rx²)`，在浮点误差下可能 `rx > 1` → `sqrt(负数)` → `NaN`。所有旋转分量前加了 `_clamp1`。

`shouldAutoDance` 的返回条件是 `mode === 'autodance' || mode === 'off'`——即使用户明确关闭了程序化动作，音频播放时也会自动切换到 AutoDance。外交官删掉了 `|| mode === 'off'`。

---

然后他打开了 `vmd-writer.ts`——因为 `procedural-motion.ts` 生成的 VMD 需要从这里写出。

第一个问题是物理尺寸。VMD 格式的尾部包含四个 uint32：camera 帧数、light 帧数、selfShadow 帧数、IK 帧数。但代码只写了三个——缺了第 4 个 IK 计数字段。babylon-mmd 的 VMD 解析器在读取尾部时会多读 4 字节，越界到下一个结构。

外交官补上了第 4 个 `setUint32`。

第二个问题是语言。VMD 格式要求骨骼名使用 **Shift-JIS** 编码。`encodeBoneName` 用的是 UTF-8。对于日文骨骼名（"センター"、"上半身"、"左腕"），UTF-8 和 Shift-JIS 的字节序列完全不同。babylon-mmd 按 Shift-JIS 解码时得到的全是乱码——骨骼名无法匹配，所有关键帧被丢弃。

外交官建了一个轻量级 Shift-JIS 映射表，覆盖了项目使用的所有日文骨骼名字符——五十音 + hiragana + 常用 MMD 汉字。未映射字符回退 UTF-8。

这解释了为什么之前程序化动作对日文模型"看起来没效果"——不是因为算法错，是因为骨骼名根本没对上。

## 织工的面料

**换装系统**（`outfit.ts` + `outfit-ui.ts`）被外交官称为"织工"——它把不同的纹理贴图换到模型的材质上，像换布料。

`_applySlot` 中的纹理加载有一个 5 秒超时。超时后，`newTex`（可能未加载完成或加载失败的 Texture）仍然被赋值到材质槽位上——覆盖了原来的纹理。如果没有加载成功，材质就变成了一片黑。

外交官改成了：加载成功（`loaded = true`）才替换；超时则 `newTex.dispose()`，保留原纹理。

`loadOutfits` 中的 HEAD 请求没有并发控制——数百个请求同时发出，浏览器限制为 6~10 个连接，其余的排队。他加了一个信号量（6 并发限制）。

**服装变体 UI**（`outfit-ui.ts`）的改造最彻底。原本的逻辑是：
- `loadOutfits` 可能抛异常或返回 null → 下一行就访问 `.variants` → 崩溃
- `applyOutfitVariant` 是异步的但没有 await → 点击后立即改变 UI，但纹理还在加载
- 重置按钮清除状态后不刷新面板 → 用户看到旧的选中标记

外交官把整个渲染逻辑封装在 `_render()` 函数内，每次操作后递归调用。所有异步操作加了 await 和 try-catch（失败时 `setStatus` 提示）。图标改用了 `createIconifyIcon`，不再用 `innerHTML` 字符串拼接。

## 声音的骨头

**摩擦**。外交官在 `xpbd-collider.ts` 的 `_solveCapsule` 中发现了一个隐秘的错误。

摩擦代码直接修改粒子的速度数组 `p.v`：
```
p.v[0] -= vtx * this.friction;
```

但 XPBD 求解器的 `step()` 末尾会用 `(p.p - p.prevP) / dt` **重新计算**所有粒子的速度——完全覆盖摩擦对 `v` 的修改。这意味着**摩擦力从未生效过**。布料在被碰撞胶囊反弹后，切线方向速度没有衰减——裙子应该黏在腿上的地方，它滑得比丝绸还顺滑。

外交官将摩擦改为基于 `prevP` 的实现：从隐含速度 `p.p - p.prevP` 中提取切线分量，衰减后写回 `prevP`。这一次，摩擦不会被覆盖。

他还注意到多个胶囊（胸、腰）可能对同一粒子重复施加摩擦——在关节区域，粒子被反复衰减切线速度，织物像被胶水粘在身体上。他加了一个 `_frictionApplied` 集合，每个粒子每帧最多摩擦一次。

## 尾声：文档的传承

凌晨两点。笔记本从第一页翻到了最后一页，又从最后一页翻到了一张新的空白页。

外交官没有写代码。他在写 `AGENTS.md`——联邦的文档宪法。

他更新了前端文件树：`core/`、`scene/`、`menus/`、`motion/`、`outfit/`、`physics/`——六个子目录，五十个文件，每个都有明确的归属。

他更新了函数映射表。物理引擎有了自己的章节：`XpbdSolver`、`createCloth`、`buildClothUpdateFn`、`SdfCollider`。程序化动作也独立了：`generateIdleVmd`、`generateAutoDanceVmd`、`buildVmd`、`loadVPDFromBuffer`。

他在最后面加了一个新章节——"审计记录"。里面列了 12 个关键修复，和 5 条高频错误模式。第一条是：

> **SlideMenu 生命周期**：每次 `new SlideMenu()` 前必须 `oldMenu?.dispose()`，否则 keydown 监听器累积。

这是他今晚修了六次的问题。

他关上笔记本。窗外，天色从墨黑变成了灰蓝。

聚合的代价永远存在——每一个收编的城邦都带着它的边界线。但联邦学会了另一件事：**修复不是为了消除代价。修复是为了让下一个走进联邦的外交官，能在十分钟内读懂地图。**
