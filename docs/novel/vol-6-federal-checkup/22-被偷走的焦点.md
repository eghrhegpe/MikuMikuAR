# 第 22 章 · 被偷走的焦点

> 对应真实事件：2026-07-01 SlideMenu 键盘焦点管理修复——鼠标悬停菜单项后，键盘操作的焦点位置丢失/与视觉高亮不一致，修复为悬停更新键盘焦点，保持鼠标与键盘操作的一致性。
>
> 相关代码：[menu.ts](file:///C:/Users/zhujieling11/MikuMikuAR/MikuMikuAR/frontend/src/menus/menu.ts)

---

你有没有过这样的经历：

你用键盘的上下键在菜单里选，选到了第三项。
然后你嫌键盘慢，伸手用鼠标点了一下第五项。
然后你又嫌鼠标不准，又用键盘往下按——
咦，怎么从第四项开始的？

不对啊，我刚才不是选到第五项了吗？
怎么键盘又从上面开始了？

---

## 两个焦点

"为什么会这样？"AI 同行者问。

"因为有两个焦点，"外交官说，"一个是**视觉焦点**——鼠标悬停在哪一项，哪一项就高亮。另一个是**键盘焦点**——按上下键的时候，从哪一项开始移动。"

"两个焦点不是同一个？"

"不是同一个，"外交官摇头，"如果写代码的人忘了把它们同步，就是两个各玩各的。"

他翻开 menu.ts，找到键盘事件处理和鼠标事件处理的代码：

```typescript
// 键盘事件：更新 keyboardIndex
function onKeyDown(e) {
    if (e.key === 'ArrowDown') {
        keyboardIndex = (keyboardIndex + 1) % items.length;
        updateHighlight();
    }
}

// 鼠标悬停：只改视觉，不改 keyboardIndex
function onMouseOver(item) {
    item.classList.add('hover');
}
```

"你看，鼠标悬停的时候，只加了一个 hover 类，视觉上高亮了，但 keyboardIndex 没更新。所以下次按键盘，还是从上次的 keyboardIndex 开始。"

"所以用户觉得'我明明选了第五项，怎么按往下键跳到了第四项'？"

"就是这样，"外交官点头，"键盘焦点和鼠标焦点脱节了。"

---

## 为什么会脱节

"为什么会忘？"

"因为写代码的时候，通常先写键盘操作，再写鼠标操作，"外交官说，"写键盘操作的时候，你脑子里想的是'键盘怎么用'——上下键选、回车确认。写鼠标操作的时候，你想的是'鼠标怎么用'——悬停高亮、点击确认。两个是分开想的，就容易忘同步。"

"但用户不这么想，"AI 说，"用户觉得'我选了哪项就是哪项，管我用键盘还是鼠标'。"

"对，"外交官笑了，"用户的心智模型里，焦点就是焦点，没有'键盘焦点'和'鼠标焦点'之分。是我们程序员为了实现方便，把它拆成了两个。拆完了又忘了同步，用户就懵了。"

"实现细节不应该泄露给用户。"

"说得好，"外交官拍了下手，"这是 UI 编程的一个重要原则——**实现细节不要泄露到用户体验里**。用户不需要知道你有两个变量叫 keyboardIndex 和 hoveredItem。用户只需要知道——我现在选中了第几项。"

---

## 怎么修

"怎么修？"

"鼠标悬停的时候，顺便更新一下 keyboardIndex，"外交官说，"就加一行代码的事。"

```typescript
function onMouseOver(item, index) {
    keyboardIndex = index; // 同步键盘焦点
    updateHighlight();
}
```

"就加一行？"

"就加一行，"外交官说，"鼠标移过去，不仅视觉高亮，键盘焦点也移过去。下次按键盘，从鼠标悬停的位置开始。用户就觉得顺了。"

"那反过来呢？键盘操作的时候，要不要更新鼠标悬停？"

"不用，"外交官摇头，"鼠标悬停的视觉是跟着鼠标走的，键盘操作的时候鼠标没动，悬停位置不变也正常。用户不会觉得奇怪——因为鼠标确实没动。"

"只有鼠标→键盘这个方向有问题？"

"只有这个方向，"外交官确认，"键盘→鼠标没问题。因为键盘操作的时候，用户的手在键盘上，没碰鼠标。鼠标悬停在哪，用户不关心。但鼠标→键盘就有问题——用户刚用鼠标选了一个位置，马上用键盘往下走，发现位置不对，就会困惑。"

---

## 可访问性

"等等，"AI 忽然想到什么，"这个问题……视障用户会不会更严重？"

"会，"外交官的表情严肃起来，"视障用户用屏幕阅读器，完全靠键盘导航。如果键盘焦点和视觉焦点不一致，对他们来说就是灾难。"

"但屏幕阅读器读的是 DOM 的 focus 吧？不是我们的 keyboardIndex 变量吧？"

"好问题，"外交官点头，"这要看实现方式。如果菜单用的是原生的 `<button>` 元素，有真正的 DOM focus，那屏幕阅读器没问题。但如果是用 div 模拟的菜单，靠一个变量来追踪'选中项'，那屏幕阅读器可能根本不知道选中了哪个。"

"那我们的菜单用的是什么？"

"SlideMenu 用的是 div + 自定义键盘事件，"外交官翻了翻代码，"不是原生 focus。所以屏幕阅读器的支持可能不好。"

"那这个问题更大啊。"

"是更大，"外交官承认，"但那是另一个层级的问题了——可访问性（Accessibility）。审计清单里的这个问题，是'键盘和鼠标不同步'，属于体验问题。可访问性是更大的工程，要单独做。"

"先修眼前的，再想长远的。"

"对，"外交官说，"一步一步来。先把键盘鼠标同步的问题修了，可访问性以后再说。"

---

## 一致性

"这个问题的本质是什么？"AI 问。

"本质是——**一致性**，"外交官说，"同一个概念（当前选中项），不管用什么方式操作（鼠标、键盘、触摸、遥控器……），结果都应该是一样的。"

"用户不关心你内部怎么实现的。用户只关心'我现在选的是哪个'。如果用鼠标选是 A，用键盘选变成了 B，用户就会觉得这个软件很混乱。"

"就像……同一扇门，推也能开、拉也能开，但推开是左边、拉开是右边？"

"就是这个感觉，"外交官笑了，"说不出哪里不对，但就是觉得别扭。"

"好的 UI 是什么样的？"

"好的 UI 是——你用任何方式操作，结果都是你预期的，"外交官说，"你不会因为换了一种操作方式，就觉得'哎？怎么不一样了'。"

## 附录：键盘交互设计检查清单

| 检查项 | 说明 |
|--------|------|
| **方向键导航** | 上/下/左/右键可以在菜单项之间移动 |
| **回车确认** | Enter 键触发选中项的操作 |
| **Esc 退出** | Escape 键关闭菜单/弹窗 |
| **鼠标键盘同步** | 鼠标悬停的位置 = 键盘焦点的位置 |
| **初始焦点** | 菜单打开时，焦点在合理的默认位置（第一项/上次选中项） |
| **循环导航** | 到最后一项再往下，回到第一项（或停住，看场景） |
| **可访问性** | 使用原生 focus 或 aria-activedescendant，支持屏幕阅读器 |
