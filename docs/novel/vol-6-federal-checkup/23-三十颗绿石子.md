# 第 23 章 · 三十颗绿石子

> 对应真实事件：2026-07-01 低优项批量处理，涵盖代码质量、微优化、小细节等 30 项审计遗留建议中的代表性条目。
>
> 相关代码：散落各处

---

高风险三颗红石子，中优十五颗黄石子，都处理完了。
桌上还剩三十颗绿石子——低优先级。

什么是低优先级？
就是——做了更好，不做也能跑。
用户不会因为没做就骂娘，甚至不会注意到。
但做了的话，代码会更干净、性能会更好、细节会更到位。

三十颗，一颗一颗讲就太长了。
外交官把它们分成了四堆——

**类型与安全**、**架构与重构**、**UI 细节**、**功能扩展**。

一堆一堆讲。

---

## 第一堆：类型与安全

第一堆绿石子，关于 TypeScript 类型和代码安全。

**1. 未使用的 import 清理**
"有些文件里 import 了东西但没用，"外交官拿起第一颗石子，"比如 import 了某个函数，但代码改了之后不用了，import 还留着。"

"删了不就完了？"

"删是简单，但要小心，"外交官摇头，"有的 import 虽然代码里没用到，但有副作用——比如 import 了一个 CSS 文件，或者 import 了一个会自动注册的插件。直接删会出问题。"

"所以不能让 ESLint 自动删？"

"可以让 ESLint 警告，人工确认了再删，"外交官说，"安全第一。"

**2. any 类型收紧**
"代码里还有一些 `any`，"外交官拿起第二颗，"有的是历史遗留，有的是图省事。能收紧的尽量收紧，改成 unknown 或者具体类型。"

"为什么是低优？"

"因为 272 个 ESLint warning 里，真正有风险的不多，"外交官耸耸肩，"大部分是'用了 any 但实际用对了'的情况。全改要花很多时间，收益不大。先改高风险的，剩下的慢慢改。"

**3. 常量提取**
"魔数——比如 `if (x > 0.1)` 里的 0.1，"外交官拿起第三颗，"是什么意思？为什么是 0.1 不是 0.2？没人知道。提取成常量，加个注释，代码就好懂多了。"

"又是那种'说起来简单但就是容易忘'的事？"

"就是那种，"外交官笑了。

---

## 第二堆：架构与重构

第二堆绿石子，关于代码结构和架构。

**4. 大文件拆分**
"settings.ts 一千多行，model-detail.ts 也一千多行，"外交官拿起第四颗，"太大了，找东西不方便，多个 AI 同时改也容易冲突。应该拆成小文件。"

"为什么是低优？"

"因为拆文件是纯重构，功能不变，"外交官说，"用户感知不到。而且拆不好容易出 bug。优先级自然就低了。"

"但对开发者很重要啊。"

"对开发者重要，但对用户不重要，"外交官点头，"优先级排序，用户的痛点在前，开发者的痛点点在后。"

**5. 重复代码抽象**
"motion-popup 和 library 里有一些相似的列表渲染逻辑，"外交官拿起第五颗，"scene-menu 和 env-menu 里也有一些相似的滑块/开关模式。可以抽成公共组件。"

"为什么不抽？"

"抽组件需要设计 API，"外交官说，"设计得好，大家都方便。设计得不好，大家都受罪。在需求还不稳定的时候，宁可重复一点，也不要过早抽象。"

"过早抽象是万恶之源？"

"差不多，"外交官笑了，"重复可以以后再抽象。抽象错了，改起来更麻烦。"

**6. 事件中心**
"现在各个模块之间的通信，有的用自定义事件，有的直接调用，有的靠全局变量，"外交官拿起第六颗，"应该有一个统一的事件中心——模型加载完成、扫描完成、预设切换……所有事件从一个地方走。"

"那是架构级的改动了。"

"是架构级的，"外交官确认，"所以优先级低——改起来动静大，风险高，用户还感知不到。等哪天加新功能的时候再顺便做。"

---

## 第三堆：UI 细节

第三堆绿石子，关于界面的小细节。

**7. 加载状态的多样性**
"现在的加载状态就是三个字'加载中…'，"外交官拿起第七颗，"不同场景可以不一样——模型库扫描可以显示'扫描中…'，预设加载可以显示'加载预设…'，布料重建可以显示'重建布料…'。"

"用户能区分出来？"

"能，"外交官说，"虽然都是等，但用户知道在等什么，心里就有数。比干巴巴的'加载中'强。"

**8. 空状态设计**
"模型库为空的时候，显示什么？"外交官拿起第八颗，"现在可能就是一片空白，或者一行'暂无数据'。好的空状态应该告诉用户——为什么空、该怎么办。"

"'还没有模型，点击添加按钮导入模型'？"

"对，就是这种，"外交官点头，"空状态是最便宜的用户引导。"

**9. 错误状态设计**
"加载失败了显示什么？"外交官拿起第九颗，"现在可能就是静默失败，或者控制台打个 log。好的错误状态应该告诉用户——出了什么错、能不能重试、怎么解决。"

"又是那种'做了更好，不做也能跑'的事。"

"就是那种，"外交官笑了，"但这些小事加起来，就是软件的质感。"

**10. 菜单过渡动画**
"SlideMenu 的进出动画，"外交官拿起第十颗，"现在是硬切还是有过渡？如果是硬切，可以加个 200ms 的滑入滑出。"

"200ms 能有什么区别？"

"区别大了，"外交官说，"有过渡，用户感觉'流畅'。没过渡，用户感觉'突兀'。200ms，不长也不短，刚好。"

---

## 第四堆：功能扩展

第四堆绿石子，关于小功能、小扩展。

**11. 道具系统滑块配置化**
"道具系统的参数，现在是写死在代码里的，"外交官拿起第十一颗，"应该做成可配置的——大小、旋转、位置——用户可以自己调。"

"用的人多吗？"

"不多，"外交官承认，"所以优先级低。但真有人用的话，就会觉得'这个功能好糙'。"

**12. 眨眼调试提示**
"程序化动作的眨眼功能，"外交官拿起第十二颗，"现在用户不知道眨眼是不是在工作——因为眨眼本来就很快，看不出来。可以加个调试选项，或者在 UI 上显示状态。"

"这都算？"

"这算，"外交官笑了，"用户花了十分钟调眨眼参数，结果不知道有没有生效——这也是体验问题。"

**13. outfit 加载反馈**
"换装的时候，换一套衣服要等一下，"外交官拿起第十三颗，"等的过程中，UI 没有任何反馈。用户可能会再点一下，然后两套一起加载。"

"加个 loading 状态？"

"加个 loading，"外交官点头，"和 motion-popup 的加载占位是一个道理。有反馈，用户就安心。"

**14-30. 还有十六颗**

外交官把剩下的绿石子往前扒了扒：

"还有很多——快捷键自定义、多语言、主题色自定义、导出截图、录制视频、模型收藏夹、最近打开列表、批量重命名、标签系统增强、搜索历史……"

"全是低优？"

"全是低优，"外交官点头，"不是不重要，是还没到时候。核心功能稳定了，再慢慢加这些。"

---

## 低优项的意义

"三十颗绿石子，全是这种'做了更好，不做也行'的事？"AI 问。

"差不多，"外交官说，"但你别小看它们。一个软件能不能从'能用'变成'好用'，就看这些低优项做了多少。"

"什么意思？"

"高风险项决定软件能不能用——不能崩、不能丢数据、不能有安全漏洞。中优项决定软件顺不顺手——加载快不快、操作流不流畅、有没有明显的 bug。"

"那低优项呢？"

"低优项决定软件有没有灵魂，"外交官看着那堆绿石子说，"细节、质感、人情味——这些东西，单个拿出来都不起眼。但放在一起，用户就会觉得'这个软件做得很用心'。"

"用心是能感觉到的。"

"能，"外交官点头，"说不上哪里好，但就是觉得舒服。那就是这些低优项在起作用。"

---

## 后来呢

三十颗绿石子，这一批没全做。

做了多少？
做了最显眼的十几颗——未使用 import 清理了一些、常量提取了几个、加载状态加了几个、空状态加了两个、过渡动画加了一个。

剩下的呢？
剩下的，记在清单上，排在队尾。
等哪天有空了，或者哪天用户提了，再翻出来做。

软件开发就是这样的。
永远有做不完的事。
永远有排不完的优先级。
重要的是——知道什么先做，什么后做，什么可以不做。

外交官把那堆绿石子收进一个小盒子里。
盒子上写着：「以后再说」。

然后他站起身，伸了个懒腰。

"高风险三颗，中优十五颗，低优做了七八颗，"他说，"差不多了。这一轮审计收尾吧。"

"那……下一轮呢？"AI 问。

"下一轮？"外交官笑了笑，"下一轮，就该建新东西了。总不能一直修修补补吧。"

窗外的夕阳照进来，把桌上的石子染成了金色。

三颗红的，十五颗黄的，十几颗绿的。
大大小小，深深浅浅。
都是这一轮审计的收获。

审计结束了。
但故事还没有。

## 附录：优先级判断速查表

| 优先级 | 用户痛感 | 典型例子 | 什么时候做 |
|--------|---------|---------|-----------|
| 🔴 高 | 会骂娘 | 崩溃、数据丢失、安全漏洞 | 立刻做，不能等 |
| 🟡 中 | 会皱眉 | 卡顿、操作别扭、小 bug | 近期做，影响体验 |
| 🟢 低 | 注意不到 | 代码质量、微优化、小细节 | 有空就做，没空拉倒 |

> 优先级不是绝对的，会变。用户反馈多了，低优会变中优，中优会变高优。要随时调整。

---

*教训：低优项不是不重要，是还没到时候。核心功能不稳的时候，去做细节优化，是本末倒置。但核心功能稳了之后，这些小细节就是拉开差距的地方。一个软件能不能从「能用」变成「好用」，就看这些低优项做了多少。*
