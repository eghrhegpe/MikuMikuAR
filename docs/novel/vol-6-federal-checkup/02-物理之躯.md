> 对应真实事件：2026-06-30 第二波审计，覆盖 scene-lighting/lipsync/loader/material/vmd/camera/env-bridge/env-impl/env-water 以及 physics/xpbd-* 全家桶，修复约 30+ 项问题，其中 xpbd-solver 子步积分重构为物理引擎核心修正。

## 光照之城

**光照之城**（`scene-lighting.ts`）是第一批被审计的城邦之一，但在第一波中它只是被顺带看了一眼。外交官还记得当时在笔记本的角落里画了个圈，标注"阴影 bias 未生效"——但现在他发现那只是冰山一角。

真正的秘密藏在**平滑过渡**里。

`transitionLighting` 是一个两秒的灯光动画——夕阳渐变为正午，冷暖色温平滑切换。外交官看着它的代码，瞳孔微微收缩。

它每帧调用 `setLightState(shadowResolution: interpolated_value)`。在 `setLightState` 内部，每当 `shadowResolution` 发生变化，就会**销毁整个 ShadowGenerator，然后在同一帧内重建它**。而 `transitionLighting` 在两秒内会执行约 120 帧。120 次销毁+重建。

"这是 120 个新的 `ShadowGenerator`，每一个都遍历所有模型的所有 mesh 添加阴影投射者，"外交官喃喃道，"在大场景中，这足以把浏览器打到假死。"

他立刻动手修复：在动画循环中加入一个 `rebuildShadowKeys` 集合——`shadowResolution`、`shadowType`、`shadowEnabled`——动画进行中跳过这些键，只在最后一帧一次性应用。

修完后他想到了第二件事：`shadowBias`。

`setLightState` 接受的 `shadowBias` 参数被存储到模块变量 `_shadowBias` 中，但如果阴影生成器已经存在，它**不会**被应用到生成器上——因为只有重建时才会读取 `_shadowBias`。这意味着用户在 UI 上滑动 `shadowBias` 滑块时，实际上什么都没有发生。

外交官加了一行：如果生成器存在，直接更新 `generator.bias`；否则标记需要重建。

然后他注意到一个隐蔽得多的逻辑：**夜间自动关闭阴影**。代码在光线强度降到 0.1 以下时——即使用户明确打开了阴影开关——会强制把 `_shadowEnabled` 设为 `false`。这行注释写着"避免无用 GPU 开销"，但它没有检查用户是否显式设置了阴影。

外交官加了一个条件：只有在用户**未显式传入** `shadowEnabled` 时才自动关闭。如果用户说"开"，那就是开。

## 口型之城

**口型之城**（`scene-lipsync.ts`）是一个安静的城邦——它检测音频频段能量，驱动模型的嘴型 morph。它已经安静了很长时间。外交官怀疑它根本没有在工作。

他检查了核心代码，发现了原因。

```typescript
import { modelRegistry } from '../core/config';
const inst = modelRegistry.get(modelId);
```

在联邦的早期，`modelRegistry` 是 `config.ts` 中的一个空 `Map`。后来，`ModelManager` 创建了自己的内部 `Map`，然后通过 `setModelRegistry()` 将它**重新赋值**给 `config.ts` 的导出变量。这是一个巧妙的桥接——ES 模块的 live binding 确保所有导入方看到最新值。

但 `scene-lipsync.ts` 在 `ModelManager` 创建**之前**就被导入了。如果 `updateLipSync()` 在 `initScene()` 执行完之前被调用——理论上不会，但这是脆弱的时序依赖，与场景加载器一个月前踩过的同一个坑如出一辙。

外交官改为注入 `ModelManager` 实例，直接调用 `_modelManager.modelRegistry.get(modelId)`。

他还发现了一个精妙的逻辑漏洞：当用户切换聚焦的模型时，`lipSyncMorphName` 缓存着旧模型的口型 morph 名称。如果新模型恰好也有同名 morph（比如通用的日语假名 `あ`），LipSync 会在错误的模型上操作 morph 权重。外交官在 `updateLipSync` 中加了一个 `lastFocusedId` 检测——聚焦变化时自动重置。

## 加载之城

**加载之城**（`scene-loader.ts`）的问题是他最熟悉的——因为它复制了光照之城、口型之城、以及 Bug 35 的所有问题。

注册时机错误：`_modelManager.register(inst)` 发生在 VMD 加载（一个异步操作）**之前**。如果 VMD 加载失败，catch 块会手动 dispose 所有 mesh——但这些 mesh 已经被注册到 ModelManager 里了。`ModelManager.remove()` 会再次 dispose 它们。

双重销毁。

外交官把注册移到了 VMD 加载之后——但发现 `loadVMDMotion` 内部通过 `modelRegistry.get()` 查找模型，所以注册**必须**在 VMD 加载之前。他改用了另一个方案：在 catch 中通过 `_modelManager.remove(registeredId)` 统一清理，ModelManager 自己负责 dispose。如果还没注册就失败了，才 fallback 到手动 dispose。

他还发现：如果模型已经加载过，重复检测使用的是 `import { modelRegistry }` → `for...of modelRegistry` 迭代。他用 `_modelManager.findByFilePath()` 替换了它。

最后，他注意到加载锁只是一个静默的 `return`。并发请求被丢弃时，用户看不到任何反馈。他加了一行状态提示："模型正在加载中，请稍候。"

## 材质之城

**材质之城**（`scene-material.ts`）的审计更像一次体检——代码看起来干净，但外交官总觉得有什么东西在皮肤下面蠕动。

他找到了。`_capture` 和 `_applyAll` 函数将 mesh 的材质强制转换为 `StandardMaterial`：

```typescript
const m = mesh.material as StandardMaterial;
```

如果模型中存在非 `StandardMaterial`（比如 PBRMaterial），访问 `diffuseColor` 等属性会直接抛出运行时异常。虽然当前项目只用 `StandardMaterial`，但这是一个等着爆炸的雷。

外交官把所有 `as StandardMaterial` 改成了 `instanceof StandardMaterial` 守卫。

他还发现 `resetMatCatParams` 只清理了分类状态（`_catState`），没有清理逐材质覆盖（`_matState`）。这意味着用户重置分类参数后，如果再次触发任何材质更新，旧的逐材质覆盖会死灰复燃。他加了一行：`_matState.delete(id)`。

最后，他在模型加载流程中加了材质原始值预捕获——确保即使从未调整过材质的模型也能正常执行"重置"操作。

## VMD 之城与相机之城

**VMD 之城**（`scene-vmd.ts`）的审计揭示了一个累积泄漏：每次加载新动作时，旧的 WASM 动画句柄没有被释放。`setRuntimeAnimation(handle)` 直接覆盖旧句柄，但 `MmdWasmAnimation` 的 WASM 侧资源——骨骼轨道、morph 轨道——仍在 WASM 堆中占据内存。频繁切换动作（如舞蹈套装预览）会导致 WASM 内存持续膨胀。

外交官在创建新动画前加了一行：`inst.mmdModel.setRuntimeAnimation(null)`。

**相机之城**（`camera.ts`）的泄漏更加隐蔽：每次加载或清除相机 VMD 时，`MmdCamera` 被从场景的活跃相机列表中移除，但从未调用 `dispose()`。GPU 资源——渲染目标、贴图——在每次切换中累积。他加了两处 `_mmdCamera.dispose()` 调用。

## 环境之桥

**环境之桥**（`scene-env-bridge.ts`）的审计是外交官今天最复杂的修复。

这是一个承担了核心职责的文件——它连接了环境系统与灯光系统，管理时间流转、自动光照联动、预设切换。但它有两个致命问题。

第一个：**循环依赖**。`scene-env.ts` 从 bridge 导入时间流转函数，bridge 又从 `scene-env.ts` 导入 `applyEnvState`。TypeScript 能解析，但运行时可能 `undefined`。外交官把 bridge 的导入改为直接引用 `scene-env-impl` 底层模块，在 bridge 内部复制了 `_applyEnvStateFacade` 的 try-catch 包装逻辑，彻底打断了循环。

第二个：**预设动画每帧触发保存**。`applyEnvPreset` 在两秒的动画循环中，每帧调用 `setEnvState` 和 `setLightState`，这两个函数各自触发 `triggerAutoSave()`。120 帧 × 2 个保存 = 240 次 I/O。外交官在 `scene-lighting.ts` 中新增了 `setSkipLightAutoSave()` 开关，预设动画期间抑制保存，仅在最后一帧执行一次。

他还顺手清除了一个困惑他很久的逻辑：`setEnvState` 中有一大段 `isFullRestore` 检测代码，试图通过"键数 > 5 且 skyColorTop[0] === 0"判断调用是否是全量恢复。它从没生效过——它只是打印警告然后什么都不做。他删掉了那 8 行。

## 深渊之下：XPBD 物理引擎

黄昏再次降临时，外交官以为审计结束了。他错了第二次。

他在石阶上坐下，翻开笔记本，看到了最后一页——在八城审计期间他匆忙写下的一行字：

*"physics/ 目录也需要看。"*

这个目录里住着联邦最年轻的城邦：一个纯 TypeScript 的 XPBD 布料物理引擎。它不依赖 WASM Bullet，独立运行，有自己的求解器、碰撞器、布料生成器和调试渲染器。它是一周前埋下的种子。

外交官深吸一口气，打开了第一扇子城门。

### 碰撞之城

**SDF 碰撞器**（`xpbd-collider.ts`）是布料与身体交互的核心。外交官检查它的位置更新逻辑。

`SdfCollider.updateMatrices(matrices)` 接收骨骼世界矩阵数组，更新每个胶囊的 world-space 中心和方向。在 `buildClothUpdateFn` 中，这些矩阵由 `anchorMatrixFn(boneName)` 获取。

外交官停下来，看着 `buildClothUpdateFn` 的代码。

`collider.solve(solver)` 被调用了。但 `collider.updateMatrices()` 呢？

不在那里。这一行不存在。

每一个胶囊体——头、颈、胸、腰、手臂、腿——它们的初始位置在原点 `(0, 0, 0)`。之后，从来没有人更新过它们。角色走到广场的另一端，她的碰撞体还站在原点。布料穿过了身体，但物理引擎认为身体在广场入口，所以它什么都不做。

布料与身体碰撞**从未生效过**。

外交官的手在键盘上停了很久。这是一个静默的 bug——代码能跑，布料在动（被锚定骨骼拖着、被重力拉着），所以你看不出问题，除非你仔细观察裙子有没有穿过大腿。大多数用户不会。

他加上了更新逻辑：每帧从骨骼矩阵获取胶囊的世界位置，调用 `collider.updateMatrices(matrices)`，然后才执行 `collider.solve(solver)`。为了让 `SdfCapsule` 知道该查哪个骨骼，他在初始化时保存了 `boneName` 字段——它之前只存在规格表（`CapsuleSpec`）中，运行时对象里被丢弃了。

然后外交官发现了第二个 bug：摩擦失效。

`_solveCapsule` 中的摩擦代码直接修改粒子的速度数组 `p.v`：

```typescript
p.v[0] -= vtx * this.friction;
```

但 XPBD 求解器的 `step()` 函数末尾会**覆写** `p.v`——它用 `(p.p - p.prevP) / dt` 重新计算所有粒子的速度，完全覆盖摩擦对 `v` 的修改。摩擦**从未生效过**。

外交官将摩擦改为基于 `prevP` 的实现：从隐含速度 `p.p - p.prevP` 中提取切线分量，衰减后写回 `prevP`。这一次，摩擦不会被覆盖。

他还注意到多个胶囊（胸、腰）可能对同一粒子重复施加摩擦——在关节区域，粒子被反复衰减切线速度，导致布料像黏在身体上一样。他加了一个 `_frictionApplied` 集合，每个粒子每帧最多摩擦一次。

---

### 求解之核

**XPBD 求解器**（`xpbd-solver.ts`）是所有布料的物理引擎。外交官打开 `step()` 方法，看到了三部分结构：

1. 第 1 步：Verlet 积分（速度、阻尼、重力、位置预测）
2. 第 2 步：子步约束求解（4 次迭代，每次遍历所有距离/弯曲/体积约束）
3. 第 3 步：最终速度更新

看起来很标准。但他注意到一件事。

第 1 步——外力积分——使用了全局时间步长 `dt`。第 2 步——约束求解——用 `subDt = dt / substeps`。它们的时间尺度不一致。

这意味着：重力只在整帧上施加一次，约束只在子步上精细求解。增加子步数不会提升物理精度——布料的下垂速度、惯性完全由全局 `dt` 决定。子步只是在同一个"快进"位置上反复拉伸约束。

外交官坐直了身体。

这是 XPBD 的核心设计缺陷：**子步划分形同虚设**。

标准做法是：每个子步内都完成一次完整的 Verlet 积分（re-predict positions + re-apply gravity），然后求解约束，然后下一个子步。重力应该在每个子步上施加，约束应该在每个子步上修正，两者在同一时间尺度上交互。

他把整个 Verlet 积分移入了子步循环内。现在每个子步：
1. 从 `prevP` 和 `p` 计算速度（带阻尼）
2. 加 `gravity * subDt`
3. 保存 `prevP`，更新 `p += v * subDt`
4. 求解所有约束
5. 地面碰撞（加了弹性系数 `restitution`，粒子现在可以反弹了）

修复后，增加子步数——从 4 到 8 到 12——布料的行为会真正改变：更稳定的约束、更平滑的运动、更少的拉伸。

子步重新有了意义。

### 布料之身

**布料生成器**（`xpbd-cloth.ts`）是建在求解器之上的工厂。外交官发现了另一个致命问题。

在 `buildClothUpdateFn` 中，锚定粒子（裙子的最上层，绑在腰部骨骼上）被每帧传送到骨骼的世界位置：

```typescript
p.prevP[0] = p.p[0] = wx;
p.prevP[1] = p.p[1] = wy;
p.prevP[2] = p.p[2] = wz;
```

`prevP` 和 `p` 被同时设置到同一个位置。这意味着锚定粒子的速度永远是零——XPBD 求解器从 `p - prevP` 推导速度，而 `prevP = p` 时速度为零。

但角色在动——她转身、走路、跳跃。当她的腰部骨骼瞬间移动了 90 度，锚定粒子瞬移到新位置，速度为零。但第二层粒子（被距离约束连接到锚点）的 `prevP` 仍然记录着上一帧基于**旧锚点位置**的物理运动。距离约束看到一个锚点瞬间偏移了一大段距离，而相邻粒子还在原地。校正力急剧增大——布料**撕裂**。

外交官删除了 `prevP` 的覆盖。锚定粒子仍然每秒移动到骨骼的正确位置，但 `prevP` 保留了上一帧的值。求解器通过距离约束自然地传递运动——相邻粒子感受到锚点的位移，在第一层距离约束中延迟一个 subDt 得到校正，然后这是正确的物理行为。

他还注意到 `_createClothMesh` 和 `_updateClothMesh` 通过 `globalThis.BABYLON` 访问 Babylon.js。这在 ESM 环境中可能为 `undefined`——整个布料 mesh 会静默失败，创建返回 `null`，所有物理计算白算。他改为直接 `import { Mesh, VertexData, VertexBuffer, StandardMaterial, Color3, Scene }`，彻底消除了全局依赖。

同样的问题也存在于**调试渲染器**（`xpbd-renderer.ts`）——外交官顺带把它整个重写了。约束线段不再每帧销毁重建（改为复用 `LinesMesh`，仅更新顶点缓冲区），胶囊碰撞体的圆柱体旋转改用四元数精确对齐，所有 `any` 类型替换为具体的 Babylon.js 类型。

### 管理器之下

**布料管理器**（`cloth-manager.ts`）是最后一站。问题小而密集。

`toggleCloth` 只创建/销毁布料，不更新 `envState.clothEnabled`——意味着场景序列化/恢复时布料状态丢失。外交官加了一行赋值。

创建布料前没有检查是否已存在——连续点击会创建多个布料，旧的不被清理。他加了 `modelManager.clothInstances.has(id)` 守卫。

模型包围盒高度可能为零——除零导致碰撞体缩放因子变成无穷。他加了对 `modelHeight > 0.001` 的检查。

还有一处：`updateCapsuleSizes` 函数试图根据骨骼距离动态调整胶囊大小，但它传入 `getBoneMatrix` 的参数是 `capsule.name`（如 `"head"`），而不是 `capsule.boneName`（如 `"頭"`）。骨骼名才是 `getBoneMatrix` 能查到的键。这个函数从未正确工作过——它永远返回 `null`，胶囊尺寸保持在默认值。外交官改了一行，骨骼对齐修复了。

## 尾声

深夜。笔记本密密麻麻。

第一波：55 项。第二波：30+。

外交官看着 `physics/` 目录里的五个文件。这个年轻的城邦在一周前被植入联邦，像一个没有神经系统的新生儿——它有肌肉（求解器），有骨骼（碰撞器），有皮肤（渲染器），但这些部件没有真正连接在一起。碰撞胶囊从未更新位置，摩擦力从未生效，子步从未真正划分时间，布料在骨骼快速移动时被撕裂。

他想起今天下午在光照之城修的那个 `shadowBias`——用户拖动滑块，系统假装在工作。那和碰撞之城的问题本质一样：**系统不会告诉你它在骗你**。代码能跑，输出看起来差不多，所以你很难发现"差不多"里面缺了最关键的东西。

他站起来，在笔记本的最后一页写下一行字：

---

*教训：一个物理引擎能运行和它能正确模拟物理是两个完全不同的概念。*

---

*聚合悖论记录：XPBD 物理引擎收编完毕。代价——它引入了四条新的边界线：显式速度与隐含速度的冲突、碰撞体的空间同步必须与渲染帧对齐、子步划分必须渗透到外力积分层、调试可视化不能依赖全局变量。每一条边界都是未来修补的锚点。*
