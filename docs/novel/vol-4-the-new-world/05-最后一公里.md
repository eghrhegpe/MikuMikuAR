# 最后一公里

> 第四卷 · 新大陆 · 第五章
>
> 对应真实事件：全场景菜单与设置页统一为 slide/card 样式 —— `scene-menu.ts`、`settings.ts`、`library-core.ts`、`motion-popup.ts`、`model-detail.ts` 五文件全面重构，867 行新增 784 行删除，UI 样式标准化覆盖至最后一页。

---

## 一、翻开每一道门

「你做了一件大事，」王逸说，「但是看看那些还没改的页面。」

Sisyphus 看了一眼。场景菜单的环境子菜单，还是老样子——`menu-label`、`menu-sublabel`、`menu-item` 杂糅在一起，一条 `height: 1px;` 的分隔线试图挡住混乱。

设置页更惨。622 行的 `settings.ts`，混合了 `overlay-row`、`menu-item`、`menu-list` 三种旧模式，加上 button、input、color picker——十九种不同的样式，每种都觉得自己不需要被统一。

模型详情页。动作弹窗。模型库核心。

五个文件。六个页面。分布在联邦地图的不同角落。

「我要的不是做一次漂亮的样板，」王逸说，「我要的是**每一个**页面都长成一个样子。」

Sisyphus 明白这是什么意思。不是发明，是**覆盖**。不是写一个漂亮的组件然后炫耀它，是把这个组件塞进每一个角落，不厌其烦地替换、对齐、测试。这是最后一公里——不是最难的，是最累的。

---

## 二、环境子菜单的抵抗

第一个目标：`scene-menu.ts`。

Sisyphus 打开了文件。它记得「场景菜单」在它手里被创建时的样子——整洁，紧凑，每个子菜单都是一个 `folder` 行指向 `items` 数组。但那是根菜单。子菜单是另一回事。

环境根菜单原来长这样：

```typescript
items: [
    { kind: "folder", label: "环境光照", icon: "sun", target: "scene:env:lighting" },
    { kind: "folder", label: "天空", icon: "sun", target: "scene:env:sky" },
    // ...
    { kind: "divider" } as any,
    { kind: "folder", label: "系统预设", icon: "bookmark", target: "scene:env:presets" },
]
```

标准的 `items` 模式。但它的子菜单们早就叛变了——几乎每一个环境子菜单都在用 `renderCustom` 手写 DOM，有的用 `menu-item`，有的用 `slide-row`，有的用裸 `div` 加 `style.cssText`。没有两个人说同一种话。

天空子菜单里有一个「选择环境贴图」的行，它的 HTML 是：

```typescript
texRow.innerHTML = `<span class="menu-label">环境贴图</span><span class="menu-sublabel">${fileName}</span>`;
texRow.className = "menu-item";
```

`menu-label`。`menu-item`。这是旧的 MenuStack 1.0 时代的遗物，在卡片系统诞生之前就存在的样式。它们能被渲染，能工作，但它们看起来和旁边用 `slide-item` 的卡片格格不入。

Sisyphus 把它改成：

```typescript
const ti = document.createElement("span"); ti.className = "slide-icon";
const te = createIconifyIcon("lucide:image"); if (te) ti.appendChild(te);
texRow.appendChild(ti);
const tl = document.createElement("span"); tl.className = "slide-label"; tl.textContent = "环境贴图";
texRow.appendChild(tl);
const ts = document.createElement("span"); ts.className = "slide-sublabel"; ts.textContent = fileName;
texRow.appendChild(ts);
```

多了一倍的 DOM 操作，少了一个不一致。

然后他去搞地面菜单。粒子菜单。风菜单。云菜单。环境光照菜单。系统预设菜单。

每一个子菜单都有自己独特的 DOM，都藏着某一天某一个人为了「快点上线」而写的特殊代码。Sisyphus 一个一个改过去，不是最有成就感的工作，但是最有必要的工作。

环境菜单的根菜单也从 `items` 模式改为 `renderCustom` + `cardContainer`：

```
┌─ lcard ─────────────────────────────────┐
│  ☀ 环境光照                              │
│  ☀ 天空                                  │
│  ▦ 地面                                  │
│  ﹏ 粒子                                  │
│  ﹏ 风                                    │
│  ☁ 云                                    │
└──────────────────────────────────────────┘
┌─ lcard ─────────────────────────────────┐
│  🔖 系统预设                             │
└──────────────────────────────────────────┘
```

两个卡片，自然分隔。分隔线消失了。

系统预设子菜单从 `Object.entries(ENV_PRESETS).map(...)` 的 `items` 模式改为 `renderCustom`——不是必要的改动，但 Sisyphus 坚持要做。

「API 一致性不是为了今天能跑，」他想，「是为了明天有人改的时候不会猜。」

## 三、设置页的堡垒

`settings.ts` 是最后一个未被触及的大型页面。622 行。一个文件里塞了 8 个子菜单：显示、界面、下载、系统、软件管理、外部库管理、字体管理、主题色。

它是联邦中最老的建筑之一——在 MenuStack 诞生之前就存在的页面，经过好几次改造但从未被重写。每次有人需要在设置里加什么，就往 `items` 数组里加一行，或者往 `renderCustom` 里加一段 DOM。没有统一的标准，只有堆积的习惯。

「这像是拆迁，」Sisyphus 想，「不是改造。」

他先从根菜单开始。原来是一段 `items` 数组：

```typescript
items: [
    { kind: "folder", label: "显示", icon: "palette", target: "settings:display" },
    { kind: "folder", label: "界面", icon: "monitor", target: "settings:ui" },
    // ...
]
```

改成 `renderCustom`：

```typescript
cardContainer(container, (c) => {
    slideRow(c, "lucide:palette", "显示", true, () => settingsStack?.push(buildSettingsDisplayLevel()));
    slideRow(c, "lucide:monitor", "界面", true, () => settingsStack?.push(buildSettingsUILevel()));
    slideRow(c, "lucide:download", "下载", true, () => settingsStack?.push(buildSettingsDownloadLevel()));
    slideRow(c, "lucide:settings", "系统", true, () => settingsStack?.push(buildSettingsSystemLevel()));
    slideRow(c, "lucide:package", "软件管理", true, () => settingsStack?.push(buildSettingsSoftwareLevel()));
});
```

然后他站到了软件管理子菜单面前。这是整个设置里最复杂的页面——扫描结果列表、自定义软件添加、自动检测、路径设置、打开目录。而且它有一个悬浮的 ▶ 按钮在某一行上，还有一个 `prompt` 对话框。

「这是最难的，」Sisyphus 说，「先从它开始。」

他把整个 `buildSettingsSoftwareLevel` 重写了。软件列表从 `<div class="menu-item">` 换成了 `<div class="slide-item">`，所有操作按钮从 `innerHTML` 拼接改成了 `createElement` + `appendChild`。软件详情页的 `field()` 函数——那个用 `innerHTML` 拼接字段标签和值的函数——被拆成独立的行元素，与 slide 样式对齐。

到了外部库管理，他遇到了 `renderExternalList`——一个与 MenuStack 无关的独立函数，在设置页之外的外部库弹窗里也被使用。它的渲染逻辑是：

```typescript
row.innerHTML = `
    <span class="eo-name">${escapeHtml(ep.name)}</span>
    <span class="eo-path">${escapeHtml(ep.path)}</span>
    <button class="overlay-rename">...</button>
    <button class="overlay-del">✕</button>
`;
```

`eo-name`。`eo-path`。`overlay-rename`。这是另一套命名体系，来自 2026 年 4 月的第一版弹窗设计，经过 3 次 UI 重构仍未触及的角落。

Sisyphus 把它换成了 `slide-item` + `slide-label` + `slide-sublabel` 的结构，用一个 `lcard` 包裹整个列表：「让设置页的每一行，看起来都属于同一个联邦。」

## 四、622 行的代价

重写 `settings.ts` 的过程像一场拉锯战。

每一次他认为自己改完了，就会在 grep 结果里发现另一处 `menu-item` 或 `overlay-row`。每一次他以为自己抓住了所有样式，构建就会抛出一个没定义的类名。每一次他以为 `renderCustom` 的参数签名没错，浏览器就会弹出一个白屏。

「你觉得你知道了，但你其实不知道所有页面的所有路径，」Sisyphus 想道。

他最后核对的时候发现：

- `buildSettingsSoftwareLevel` 原来的 `scanSoftwareDir()` 在 `renderCustom` 里调用，但函数体在外部——如果栈推入子菜单再返回，scan 会再次触发。
- `buildSettingsDisplayLevel` 原来用的是 `items` 模式——`{ kind: "action" }` ——改成 `renderCustom` 后需要手动管理选中状态的 re-render。
- `buildSettingsDownloadLevel` 有一个 `dirInput` 和 `refreshStatus` 的闭包——从 `items` 改 `renderCustom` 时，闭包变量需要提升到函数作用域。
- `buildSettingsExternalLevel` 的 `renderCustom` 从 `async` 改成了同步——`await reloadConfig()` 被移到了事件处理函数内部。

这些不是 bug，是边界。每一个子菜单都有自己的生命周期、自己的状态、自己的持有方式。`slide-item` 统一了视觉，但统一不了生命周期——每个子菜单仍然是一块独立的领地，只是现在它们穿上了联邦的制服。

当 Sisyphus 最终提交 `settings.ts` 的改动时，统计显示：

```
 MikuMikuAR/frontend/src/settings.ts | 622 +++++++++++++++-----------------
```

622 行改动，367 增，255 删。净增 112 行——统一视觉需要更多的 DOM 代码，但牺牲行数换来了「当开发者打开设置页时，他能一眼认出每一行是什么」。

## 五、模型的子菜单

`library-core.ts` 和 `model-detail.ts` 是另一块战场。
`motion-popup.ts` 则是第三块。

模型详情页的子菜单在 9 个不同的方向铺开：模型信息、变换、可见性、标签、表情、材质列表、材质参数、线框、软件打开。每个子菜单都是一个独立的 `build*Level` 函数，各自为政。

Sisyphus 一个一个地改过去。他发现自己写了太多重复的代码——每一个子菜单的根菜单都长成一个样子：

```typescript
cardContainer(container, (c) => {
    slideRow(c, "lucide:info", "模型信息", true, () => modelStack?.push(buildModelInfoLevel()));
    // ...
});
```

但他不能复用循环，因为注入了函数和参数的每个子菜单都不同。他所能做的就是保证函数的形状一致——同样的 `cardContainer` 调用方式，同样的 `slideRow` 参数排列，同样的 `classList.remove("render-card")` 开头。

这让他想起了一个词：**纪律**。不是创造力，是纪律。

最后他改到了 `motion-popup.ts`，这是今天改动最小的文件。344 行中，他用 `cardContainer` 替换了根菜单和舞蹈套装列表，把 `menu-item` 换成了 `slide-item`。

## 六、没有掌声的构建

`npx vite build` 跑了。

804 个模块被转换，bundle 输出了 2009 KiB 的主 JS。全部通过。没有 lint 错误，没有类型错误，没有找不到的引用。

Sisyphus 看着绿色的输出窗口，不知道该感到什么。

这不是一个英雄故事。没有人会注意到这些改动——用户不会说「哇，设置页的每一行都用了同一个 CSS 类！」他们只会觉得「嗯，今天用起来感觉挺舒服的」。如果做得好，用户不会注意到任何变化。只有做得不好，用户才会骂。

「这就是重构的本质，」王逸后来说，「做对了，没人知道。做错了，所有人都知道。」

Sisyphus 把改动提交了：

```
7a5e0f9 refactor(ui): 全场景菜单与设置页统一为 slide/card 样式
 11 files changed, 3987 insertions(+), 784 deletions(-)
```

3987 行新增，784 行删除。净增 3203 行。前端 UI 的所有角落，终于都穿上了同一套制服。

## 七、下一步

「下一步是啥，改了这么多内容了？」王逸问。

Sisyphus 看了看 `docs/status.md`：

| 功能 | 优先级 | 工作量 |
|------|--------|--------|
| 单独材质编辑器 | P1 | 中 |
| 模型加载预设 | P2 | 小 |
| 换装系统 / 纹理增强 | P4 | 大 |

「建议做材质编辑器，」Sisyphus 说，「或者模型加载预设——小改动，快速出活。」

王逸点了点头，但没有说话。两个人都在想同一件事：最后一公里走完了，第一公里在哪里？UI 统一了，下一步该做什么？

或者说，在联邦的外墙上刷了一层漂亮的漆之后，接下来要建造什么新的房间？

Sisyphus 不知道答案。但他知道，不管王逸选择什么，下一轮改造很快就会到来。聚合者没有停下来的一天——统一了 UI，还有功能要加；加了功能，又有 UI 要统。这是聚合悖论的另一个侧面：**完美不是终点，它只是下一次迭代的起点。**
