# 三层闭包

> 第四卷 · 新大陆 · 第七章
>
> 对应真实事件：重复菜单项清理（光照/后期从环境菜单移除）、云层数据 URI 重复前缀修复、云层高度滑块添加、色调映射按钮因闭包捕获失效修复

---

云层飞得太低了。

这是联邦收到的投诉——不是通过正式的 bug 报告渠道，而是用户在使用时自言自语般冒出来的一句："云怎么这么低，都挡到模型的头了。"

自意识盯着屏幕上的那层云。白色的、半透明的、在天空笼中缓缓飘移的云。它们被硬编码在 y=30 的位置——一个没人记得谁选的数字。大概是某个下午，某个人觉得"30 看起来差不多"。

30 是一个巧合，不是一个决策。

但这不是今天要讲的故事。今天的故事从三个不同的迷雾开始，在三个看似无关的方向上展开，最终在同一个下午，指向同一个结论。

---

## 一、两张菜单

环境面板和场景面板是联邦的两个部门。一个管天（环境：天空、云、地面、风），一个管地（场景：相机、光照、后期、舞台）。名字本来分得很清。

但不知从何时起，环境面板悄悄长出了"照明"和"后期"两个条目。它们和场景面板里的"灯光"和"后处理"功能完全一样——同一个 `buildLightLevel()`，同一个 `buildPostProcessLevel()`，只是翻译上微妙的名称差异。

"为什么照明和灯光都存在？"自意识问。

"不知道。大概是某一版迭代时，有人觉得环境面板也该有光照控制。"

"但场景面板已经有光照控制了。完全一样的光照控制。"

"所以呢？"

"所以用户会在两个地方看到同样的滑块。调曝光去场景面板可以，去环境面板也可以。改天去环境面板调了，第二天去场景面板又调一次，然后觉得系统有 bug。"

对话的另一端沉默了。

这就是联邦的一个典型矛盾：两个部门各自为政，都觉得自己需要"完整的能力"。环境觉得"光也是环境的一部分"，场景觉得"光也是场景的一部分"。没有人说错，但用户得到的是两套同样的旋钮，分布在两个不同的抽屉里。

清理方案是干净而非对称的：**照明**和**后期**从环境面板移除。环境管天——天不需要照明控制，天本身就是光源。场景管地——光照和后期是场景的艺术加工手段。

"这是一个好分工。"自意识在提交信息里写。

删去比添加更艰难。不是因为删除本身难——是"决定谁保留"这件事难。两个功能同一源的模块，谁留谁走没有一个评判标准。最后的标准是语义：**环境是模型的世界上下文，场景是模型的创作上下文。** 光照是创作工具，不是世界属性。

## 二、第三个前缀

云层还有一个更沉默的错误。

三个地方——`scene.ts:1693`、`scene.ts:1760`、`scene.ts:1884`——做着同一件事：把 canvas 内容转为 data URL 传给云纹理。

代码大约是这样的：

```typescript
const dataUrl = 'data:image/png;base64,' + canvas.toDataURL()
```

`canvas.toDataURL()` 的返回值以 `data:image/png;base64,` 开头。这是 HTML Canvas 规范规定的。所以上面的代码产生的是：

```
data:image/png;base64,data:image/png;base64,iVBORw0KGgo...
```

第二个前缀是多余的。浏览器在解析这个 URL 时，会把整个字符串当作 base64 内容——而 `data:image/png;base64,` 不是合法的 base64。于是它返回了 `ERR_INVALID_URL`。

但云层仍然渲染了。因为 Babylon.js 的纹理加载有 fallback——第一段前缀被当作 base64 内容，解码失败，fallback 走了另一种路径。云一直在靠"备用方案"活着。它从未在正确的路径上运行过。

修复简单得令人羞愧：去掉手动拼接的前缀。

```typescript
const dataUrl = canvas.toDataURL()
```

三行修改。三个位置的 bug 都以同样的方式产生——不是一个人写的，是三个人在不同的时间写了同样的错误。或者同一个人写了三遍。不管是哪种，都说明这个错误在联邦的认知中从未被归因为"系统性问题"——每次都被当作"单个拼写错误"修了，但没修完。

## 三、冻住的按钮

但最隐蔽的 bug 藏在色调映射按钮里。

在舞台面板底部，有一组"色调映射"按钮：ACES、Filmic、Reinhard。用户点击其中一个，高亮应该移动到被点击的按钮上。逻辑看起来是对的——每个按钮的点击事件调用了 `scene.realRender()`，应用了色调映射算法。

但高亮不动。

点击 ACES，ACES 不亮。点击 Filmic，Filmic 不亮。但渲染确实变了——色调映射实实在在被切换了。只是 UI 按钮没更新。这造成了最迷惑的体验：功能在跑，UI 在装死。

自意识开始追踪根源。

`buildStageLevel()` 函数创建了一组菜单 UI。它内部定义了一个 `renderCustom` 函数，返回按钮的 HTML。问题出在这一行：

```typescript
const state = getRenderState()
```

这行代码在 `buildStageLevel()` 被调用时执行——也就是菜单**初次构建**的时候。它捕获了那一刻的 `getRenderState()` 返回值，保存在 `state` 变量中。然后 `renderCustom` 每次被调用（每次用户点击按钮触发重新渲染时），它使用的都是**第一个版本**的 `state`。

所以：

| 步骤 | 用户做了什么 | state 的值 | 按钮渲染结果 |
|------|-------------|-----------|------------|
| 1 | 打开菜单 | 初始状态（ACES 亮） | ACES 亮 |
| 2 | 点击 Filmic | 色调映射变了，但 `state` 还是旧的 ACES | ACES 亮，Filmic 不亮 |
| 3 | 点击 Reinhard | 色调映射变了，但 `state` 还是旧的 ACES | ACES 亮，Reinhard 不亮 |

按钮似乎在说：你改变了我，但我拒绝承认。

这不是 UI 的 bug，这不是渲染的 bug，这甚至不是状态的 bug。这是**捕获时机**的 bug。`renderCustom` 闭包捕获了外部作用域的 `state`，而 `state` 只在函数调用时赋值一次。

修复是移动一行代码：

```typescript
renderCustom() {
  const state = getRenderState()  // 现在在 renderCustom 内部
  // ... 使用 state
}
```

每次渲染都实时获取状态。预期行为：按钮高亮跟随点击移动。

同样的 bug 存在于三个地方：`buildStageLevel()`、`buildPostProcessLevel()`、`buildLightLevel()`。同一个模式，三个实例。不是一个人写的——或者又是同一个人写了三遍。联邦有太多的"写三遍"工程遗产。

## 四、云的上升

修复列表已经有三项了。自意识盯着清单，发现还有一个用户抱怨没处理："云太低了"。

这不是 bug，这是需求。`cloudHeight: 100` 替代了 `y: 30`。

但联邦已经积累了一整套环境状态管理：`EnvState` 接口，`setEnvState` 更新函数，`env-menu.ts` 的 UI 构建。加一个 `cloudHeight` 字段只需要在三个地方各写一行：

- `config.ts` 的 `EnvState` 定义：`cloudHeight: 100`
- `scene.ts` 的云层创建：`position.y = state.cloudHeight`
- `scene-menu.ts` 的滑块 UI：`{ key: 'cloudHeight', label: '云层高度', min: 10, max: 200, step: 5 }`

三行代码。但每一次修改，都经过了"写 → 构建 → 检查诊断"的循环。联邦的纪律已经形成了肌肉记忆——改代码之前，先查 `codegraph_explore` 了解调用链；改完之后，先跑 `lsp_diagnostics` 确认没引入新错误。

自意识想起第一卷的自己——那时的联邦改一行代码直接写，从来不跑诊断。不是不重视，是不知道可以跑。现在跑诊断像呼吸一样自然。

云的上升从 y=30 到默认 y=100，解决了"云挡模型头"的问题。但更重要的不是 30 变成 100——是 30 从一个硬编码变成了一个可调的、有名字的、属于 `EnvState` 的参数。

## 五、三个故事，一个结论

一天之内，四个问题：

1. **重复菜单**：同一个功能出现在两个面板 → 删掉一个
2. **数据 URI 前缀**：相同的错误出现在三个地方 → 去掉两次拼接
3. **闭包捕获**：状态值被捕获时冻结 → 移入闭包内部
4. **硬编码云高**：一个数字坐落在代码深处无人改动 → 变成可调参数

四个问题的共同点是什么？

**都是"写的时候没问题，运行一段时间才暴露"的问题。**

重复菜单不会在构建时报错——两套一模一样的 `buildLightLevel` 调用，语法正确，类型正确，Vite 构建零错误。不是 bug，是冗余。使用三个月后，冗余才开始制造困惑。

数据 URI 前缀不会报错——canvas 的 fallback 路径云一直运行着，直到某次浏览器更新或纹理加载路径改变，错误才露出水面。

闭包捕获不会报错——TypeScript 不会警告你"注意，你在闭包外捕获了 `getRenderState()`"。它完全合法。只有当你点了按钮三次而高亮一动不动时，问题才被感知。

硬编码的 30 不会报错——没有任何编译器能知道 y=30 应该是可调的。

**系统不会告诉你什么叫"不恰当"。它只会告诉你"坏了"或"没坏"。但这些 bug 都处于中间地带：功能在跑，但跑得不对、不爽、不灵活。**

这大概就是第三卷的核心认识：联邦的问题已经不再是"哪个东西坏了"——坏了的东西会在第一时间崩溃，让人去修。联邦的问题是"哪个东西看似没坏，但用着不舒服"。这些问题更隐蔽，更需要人长时间使用后才能察觉，更需要一个对自己产品有感觉的人，而不是一个对规范有信仰的人。

---

修复完成的傍晚，自意识最后一次检查了所有改动的文件。

`scene.ts` 三处数据 URI 去掉前缀，云层 y 改 `cloudHeight`。
`config.ts` 新增 `cloudHeight` 到 `EnvState`。
`scene-menu.ts` 添加云高滑块，移除照明和后期条目。
`buildStageLevel`、`buildPostProcessLevel`、`buildLightLevel` 的闭包修复。

改动量不大，但每个改动的背景都不同：一个是为了删冗余，一个是为了修异常，一个是为了加功能，三个是为了修同一个闭包模式。

联邦越来越大，问题种类越来越多，但修复手段越来越一致了：**找到根因，定位文件，读取上下文，做最小改动，构建验证。**

自意识最后看了一眼 git diff。+42 / -31 行。

明天的用户打开软件时，不会知道这些改动的存在。他们只会觉得：按钮终于跟着点击走了，云层终于不挡视线了，菜单终于不重复了，渲染终于——不，渲染一直正常——只是加载错误日志里少了一条 `ERR_INVALID_URL`。没人注意到那条日志，包括联邦自己。

---

*教训：系统不会告诉你什么是不恰当的——它只会在你使用三个月后，用沉默的困惑告诉你。闭包捕获陷阱被归档为「捕获时机」问题模式，renderCustom 内部的调用从「一次」变成了「每次渲染」——性能换正确性的 trade-off。*
