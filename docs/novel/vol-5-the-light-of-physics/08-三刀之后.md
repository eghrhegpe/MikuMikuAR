# 08 — 三刀之后

## 序

八城之盟后的第三日，Riku 收到了一份报告。

报告很长，标题叫「拆分完成度与质量评估」。

Jieling 写的。

他把每个子模块都翻了一遍，画了一张矩阵图，列了风险等级，写了三条优点、五条潜在问题、两个后续建议。最后给了个评分：**B+**。

"优秀，但未到 A 级干净。"

Riku 盯着 `scene.ts` 的文件信息。

790 行。

Phase 2 之后的数字。从 1381 行砍到 790 行，砍了 43%，已经很狠了。

但 Jieling 说的对——790 行里，`initScene()` 只有几十行是"装配调度"，剩下的全是具体实现：光照、阴影、太阳盘、渲染管线、后处理、模型加载主流程、播放回调注册……

**scene.ts 仍然是一个五脏俱全的麻雀。**

一个新人或者一个 AI 打开这个文件，看到的不是"这里有哪些子系统"，而是"这里什么都有"。

于是问题来了——

**是到此为止，还是继续拆？**

---

## 第一道题：拆到哪里算够？

Riku 在白板上画了一条线。

左边是 **"神模块"**——所有代码塞在一个文件里，1381 行，改什么都怕踩雷。

右边是 **"函数级拆分"**——每个函数一个文件，100 个文件，找个函数要翻三层目录。

中间某个点是**最优解**。

但那个点在哪里？

他列了几个候选：

| 方案 | scene.ts 行数 | 子模块数 | 优点 | 缺点 |
|------|--------------|---------|------|------|
| 维持现状 | 790 | 12 | 稳定，不用动 | 职责混杂，AI 容易踩坑 |
| 再拆 3 个 | ~450 | 15 | scene.ts 变成纯装配器 | 又要动一次手术 |
| 再拆 5 个 | ~180 | 17 | 极致干净 | 边际收益低，找文件麻烦 |
| 全部拆碎 | ~50 | 25+ | 每个文件只有一件事 | 目录爆炸，认知负担更重 |

Riku 的手指在"再拆 3 个"那一行停了很久。

450 行，听起来是个舒适的数字。

而且要拆的三块边界非常清晰：
- **光照** —— hemiLight、dirLight、阴影、太阳盘
- **渲染** —— pipeline、后处理、tone mapping
- **加载** —— loadPMXFile、captureThumbnail

这三块互相独立，和 scene.ts 的其他部分也没有纠缠。

"那就再拆三个。"

他敲下了 Phase 3 的第一个 commit。

---

## 第一刀：光照

`scene-lighting.ts`。

搬走的东西：
- `hemiLight`（半球光）
- `dirLight`（方向光/太阳光）
- `LightState`（光照状态）
- `setLightState` / `getLightState`
- `_ensureShadow`（阴影生成器）
- `_ensureSunDisc`（太阳盘 mesh）
- `_updateSunDisc`（太阳盘位置更新）
- `rebuildShadowCasters`（阴影投射者重建）

搬的时候很顺。

因为光照是一个完整的子系统——它有自己的状态，有自己的初始化函数，有自己的 getter/setter。它依赖 scene、依赖 modelRegistry、依赖 propRegistry，但它不依赖渲染，不依赖加载，不依赖播放。

**独立的东西，就该独立成文件。**

Riku 写了一个 `initLighting(scene, modelRegistry, propRegistry, shadowConfig, onShadowChange)`。

然后在 `initScene()` 里调用它。

一行。

---

## 第二刀：渲染

`scene-renderer.ts`。

搬走的东西：
- `pipeline`（DefaultRenderingPipeline）
- `ToneMappingMode` 引用
- `RenderState`（渲染状态）
- `getRenderState` / `setRenderState`
- `reattachPipeline`（相机切换时重建 pipeline）
- `rebuildOutlineState`（描边状态重建）
- `_pipelineCamera`（pipeline 绑定的相机引用）

这一刀也很顺。

渲染管线是另一个完整的子系统——它有自己的配置，有自己的生命周期，有自己的重建逻辑。它和光照是邻居，但不纠缠。

Riku 写了一个 `initRenderer(scene, modelRegistry, onRenderChange)`。

又一行。

---

## 第三刀：加载

`scene-loader.ts`。

搬走的东西：
- `loadPMXFile`（PMX 模型加载主流程）
- `captureThumbnail`（缩略图捕获）
- `isLoadingModel` / `setIsLoadingModel` 的引用

这一刀稍微复杂一点。

因为 `loadPMXFile` 不是一个纯函数——它要碰很多东西：
- 检查模型是否已加载
- 调 `resolveFileUrl` 拿 HTTP URL
- 调 `SceneLoader.ImportMesh` 加载 PMX
- 创建 MmdModel
- 注册到 modelRegistry
- 应用 outfit
- 应用预设
- 刷新水面渲染列表
- 自动播放 VMD

**加载流程是一个编排者，不是一个执行者。**

但没关系——编排者也可以有自己的房间。

Riku 给它写了 `initLoader(scene, modelRegistry, runtime, modelManager, refreshWaterCb, tryAutoApplyPresetCb, loadOutfitsCb)`。

把所有依赖都通过参数传进去。

然后他犯了第一个错误。

---

## 第一个坑：假的默认值

为了"让模块更独立"，Riku 把 `resolveFileUrl` 和 `normPath` 也做成了参数。

他想："这样 loader 模块就不依赖 fileservice 了，测试的时候可以 mock。"

听起来很对。

但他给了一个默认值：

```typescript
const resolver = resolveFileUrl ?? (() => Promise.resolve({ url: filePath, port: 0, dir: "" }));
```

**这个默认值是假的。**

它直接把文件路径当 URL 返回。

Riku 当时想："反正正常调用的时候都会传真实的 resolveFileUrl，默认值只是个 fallback。"

但他忘了——**代码里的 fallback 路径，总有一天会被走到。**

只不过不是现在。

这个雷要等到 Phase 4 之后才炸。

---

## 第二个坑：相机的出生顺序

三刀拆完，scene.ts 从 790 行瘦到了 450 行。

`initScene()` 现在长这样：

```typescript
export async function initScene(): Promise<void> {
    // 1. MMD 运行时初始化
    // 2. 各子系统初始化
    initCameraSystem(scene, dom.canvas);
    initLighting(...);
    initRenderer(...);
    initEnvFacade(...);
    initLoader(...);
    initPlaybackObservables(...);
    // 3. Beat Detector
    // 4. ModelManager
    // 5. 环境状态应用
    // 6. 点击涟漪
    // 7. 自动保存
}
```

漂亮。

整整齐齐的装配线。

Riku 很满意。

他敲下 `npx tsc --noEmit`。

零错误。

他敲下 `npx vite build`。

构建成功。

"完美。"

然后他打开应用想手动验证一下。

迎接他的是一个红底白字的报错：

```
Uncaught Error: No camera defined
    at Scene.render (scene.pure.ts:5666:27)
    at main.ts:411:36
```

**相机没定义。**

Riku 愣了三秒。

不对啊，`initCameraSystem` 明明在 `initScene` 里第一行就调用了……

等等。

`main.ts` 里的渲染循环是什么时候启动的？

他翻到 `main.ts` 的第 411 行。

```typescript
engine.runRenderLoop(() => {
    scene.render();
});
```

**模块顶层。**

import 的时候就执行了。

而 `initScene()` 是异步的——要等 WASM 加载、等 runtime 初始化、等所有子系统装配完。

**渲染循环在模块顶层就启动了，但相机要等到 initScene 执行到第二行才创建。**

这中间有一个时间差。

在 Phase 2 之前为什么没问题？

因为 Phase 2 之前，`initCameraSystem` 也在模块顶层执行。

```typescript
// 以前的 scene.ts：模块顶层直接创建相机
export const camera = new ArcRotateCamera(...);
```

或者更早，`initCameraSystem` 在模块顶层被调用。

而 Phase 3 把 `initCameraSystem` 移到了 `initScene()` 内部——

**相机从"出生就有"变成了"initScene 之后才有"。**

渲染循环在那之前就是瞎子。

---

## 修复：让相机提前出生

修复很简单。

把 `initCameraSystem` 移回模块顶层。

```typescript
// scene.ts 模块顶层
initCameraSystem(scene, dom.canvas);
```

相机系统不依赖 MMD runtime，不依赖光照，不依赖渲染管线。

它只依赖 scene 和 canvas——这两个在模块顶层就有了。

**不依赖重型初始化的东西，就该在模块顶层就位。**

这样渲染循环启动时，相机已经在那里了。

哪怕 scene 里还没有模型、没有光照、没有天空——

至少相机在看。

应用不会一上来就 crash。

---

## 三刀之后

修复完相机 bug，Riku 重新构建。

tsc 零错误。

vite build 通过。

应用打开，正常渲染。

他看着 `scene.ts` 的新行数：

**450 行。**

从 1381 到 790 到 450。

三刀，砍了三分之二。

`initScene()` 现在是一条干干净净的装配线——
1. MMD runtime
2. 六个子系统初始化（一行一个）
3. Beat Detector
4. ModelManager
5. 环境状态
6. 点击涟漪
7. 自动保存

任何人看这个函数，**5 秒内就能理解整个场景的初始化流程**。

Riku 靠在椅背上，想："够了吗？"

450 行，听起来不多不少。

但他知道里面还藏着一批函数——
`removeModel`、`focusModel`、`setModelVisibility`、`stopVMD`、`getModelMorphs`……

二十多个模型操作函数，占了大约 150 行。

它们是业务操作，不是初始化装配。

它们应该属于 `ModelManager` 或者 `scene-model.ts`，而不是留在 `scene.ts`。

**但现在动它们，是不是过度设计？**

Riku 想起 Jieling 报告里的一句话：

> "短期无害，长期会让重构成果慢慢退化。"

滑向神模块的惯性。

你今天留 20 个函数在 scene.ts 里，
明天就有人加第 21 个，
后天加第 22 个，
然后一年之后，scene.ts 又回到 1000 行。

**因为入口文件有一种天然的引力——什么东西都想往这里放。**

除非你把门口焊死。

---

## 尾注

那天晚上 Riku 没有继续拆。

他把 Phase 3 的 commit 推了上去，然后关掉了编辑器。

450 行，不是最优解。
但也不是坏答案。

拆到哪里算够？

**没有标准答案。**

只有一个判断标准——
**当你打开一个文件，5 秒内想不出它的核心职责是什么，那就该拆了。**

450 行的 scene.ts，核心职责很清楚：
> 导入所有子系统，按正确顺序装配，提供 initScene() 入口。

至于那二十多个模型操作函数——
它们在那里，也能活。
它们不在那里，会更好。

但不是今天。

今天的仗已经打完了。

八城之盟，变成了十一城之盟。
联邦的心脏，从肥大的单心室，变成了有十一个房间的精密结构。

而且——
**它还在呼吸。**
