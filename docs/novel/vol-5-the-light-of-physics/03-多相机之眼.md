# 03 — 多相机之眼

## 序

相机是观众的眼睛。

MikuMikuAR 支持三种相机模式：Freefly（自由漫游）、One-shot（单拍构图）、Concert（演唱会模式）。逻辑都在 `camera.ts` 里，但菜单里只有 Freefly 有参数面板，One-shot 和 Concert 像是被遗忘的房间——门存在，但没装把手。

Riku 决定给它们装把手。

---

## Freefly 的滑条

Freefly 是最常用的模式。用户可以拖动滑条调整相机距离、高度、旋转速度。

但有个小毛病：切换回 Freefly 时，菜单里的滑条显示的是默认值，不是当前值。比如用户把距离调到 15，切到 One-shot 再切回来，滑条显示 5（默认），但实际相机距离是 15。这会让用户以为参数被重置了，然后重新调一遍。

bug 很好修。`renderFreeflyParams()` 里读的不是实时值，而是初始化时的局部变量。改成从 `camera.ts` 的导出函数读取就行：

```typescript
function renderFreeflyParams(container: HTMLElement) {
    container.innerHTML = "";
    const params = getFreeflyParams(); // 新增导出函数
    addSliderRow(container, "距离", params.distance, 1, 30, 0.5, (v) => {
        setFreeflyParams({ distance: v });
    });
    // ... 其他滑条
}
```

`getFreeflyParams()` 在 `camera.ts` 里加了三行：

```typescript
export function getFreeflyParams(): FreeflyParams {
    return { ...freeflyParams };
}
```

改完，切相机模式再切回来，滑条乖乖显示当前值。

## One-shot 的构图

One-shot 模式的核心思想是：预设几个「构图」，每个构图对应一组相机参数（距离、高度、角度、FOV）。切换到 One-shot 时，相机平滑过渡到当前构图的参数。

这个功能在 `camera.ts` 里已经实现了，但菜单里没有入口。`buildCameraLevel()` 里只有 Freefly 的参数面板，One-shot 连个影子都没有。

Riku 在 `scene-menu.ts` 的相机菜单里加了 One-shot 子菜单：

```typescript
{ kind: "folder", label: "单拍构图", icon: "lucide:camera", target: "scene:camera:oneshot" }
```

然后在 `onFolderEnter` 里加了 case：

```typescript
case "scene:camera:oneshot": return buildOneShotLevel();
```

`buildOneShotLevel()` 里列了八个预设构图：全身、半身、特写、低角度、高角度、侧面、背面、俯视。每个都是按钮，点一下调 `switchCameraMode("oneshot", { preset: "full-body" })`。

Riku 试了一下。「全身」——相机后退，模型完整入框。「特写」——相机拉近，只框住脸部。「低角度」——相机压低，模型显得更高。

比他预想的顺手。

## Concert 的暂停

Concert 模式是给「演唱会场景」设计的：相机沿着预设路径运动，像现场导播在切机位。

但有个问题——一旦开始，停不下来。没有暂停按钮，没有恢复按钮。如果想在某个机位多停留一会儿，只能等下一圈。

Riku 在 `camera.ts` 里加了 `_concertPaused` 标志和 `toggleConcertPause()` 函数：

```typescript
let _concertPaused = false;

export function toggleConcertPause(): void {
    _concertPaused = !_concertPaused;
    // 暂停时冻结相机路径动画
    if (_concertPaused) {
        _concertPathAnimator?.pause();
    } else {
        _concertPathAnimator?.resume();
    }
}
```

然后在 `scene-menu.ts` 的 Concert 子菜单里加了 toggle：

```typescript
addToggleRow(container, "暂停", _concertPaused, (v) => {
    toggleConcertPause();
    sceneStack?.reRender();
}, "lucide:pause");
```

暂停时，相机定格在当前位置。恢复时，从定格处继续沿路径运动。不需要重置，不需要重新构建。

## 逻辑的审查

功能写完，Riku 习惯性地做了一遍逻辑审查。不是测试（那是后面的事），而是盯着代码看，找那些「能跑但不对」的地方。

他发现了四个问题：

**1. 水下后处理的空判断**

`scene.ts` 里有个后处理 pipeline，`chromaticAberration` 是可选属性。代码里直接写 `pipeline.chromaticAberration.aberrationAmount = v`，但如果 `chromaticAberration` 没初始化，会报 `Cannot read property 'aberrationAmount' of undefined`。

修法：加空判断。

```typescript
if (pipeline.chromaticAberration) {
    pipeline.chromaticAberration.aberrationAmount = v;
}
```

**2. 水面风向的初始化**

`_createWater()` 里设置了 `windDirection`，但只在 `setEnvState` 的 observer 里更新。如果用户打开应用后先改风向、再创建水体，水体的风向是默认值，不是用户刚设的值。

修法：在 `_createWater()` 末尾加一行，用 `envState.windDirection` 初始化。

```typescript
waterMaterial.windDirection = new Vector2(
    Math.cos(envState.windDirection * DEG_TO_RAD),
    Math.sin(envState.windDirection * DEG_TO_RAD)
);
```

**3. waveSpeed 的联动**

`waterAnimSpeed` 是用户可调的「水体动画速度」。改了这个参数，代码里只更新了 `windForce`，没更新 `waveSpeed`。结果：水体动画速度快了，但波浪的物理速度没变，看起来「风大了但浪没大」。

修法：同时设置 `waveSpeed`。

```typescript
waterMaterial.windForce = envState.waterAnimSpeed;
waterMaterial.waveSpeed = envState.waterAnimSpeed * 0.8; // 经验系数
```

**4. 天空旋转的溢出**

天空盒的 `rotation.y` 每帧累加 `skyRotationSpeed * deltaTime`。如果应用开着很久（比如挂了一晚上），`rotation.y` 会变得很大，浮点数精度开始掉，天空盒抖动。

修法：加溢出保护。

```typescript
skyBox.rotation.y = (skyBox.rotation.y + delta) % (Math.PI * 2);
```

四个问题，四个修法。没有重构，没有重写，只是把裂缝填平。
