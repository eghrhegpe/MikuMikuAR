# 04 — 悬空之影

## 序

弹窗是联邦与用户对话的窗口。

五个弹窗（模型/动作/设置/场景/环境）原本分属四个独立容器，经「统一容器改造」后全部迁入 `sceneOverlay` 一室。室只有一个，内容随按钮切换而更替——这本是更简洁的架构。

但 Riku 发现了一个诡异的现象：

> 点击按钮，弹窗打开。点击画布，弹窗关闭。再次点击按钮——弹窗不出现了。

`aria-expanded` 在按钮上正常切换，CSS 的 `visible` 类也正常添加。但弹窗的内容区域是空的——好像有什么东西挡在中间，让内容永远到不了屏幕。

---

## 史官的奏报

Jieling 的描述很精确：「aria-expanded 会正常变换，感觉是时序、多个代码同时影响导致的。」

Riku 开始审计。他先假设是 `toggleOverlay` 的时序竞争——也许 `closeAllOverlays()` 清除了刚加上的 `visible` 类？

他读了三遍 `toggleOverlay` 的逻辑：

```typescript
// 打开分支
} else {
    closeAllOverlays();
    showFn();
    el.classList.remove("overlay-fade-out");
    el.classList.add("visible");
}

// 关闭分支
if (last === showFn) {
    el.classList.remove("visible");
    setPopupOpen(false);
    syncNavAriaExpanded();
    _lastOverlayFn.delete(id);
}
```

看起来没问题。`closeAllOverlays()` 清除其他弹窗的 `visible`，然后 `showFn()` 渲染内容，最后加上 `visible`。

但等等——`showFn()` 是什么？

对于模型弹窗，`showFn` 是 `showModelPopup()`。Riku 打开了 `library-core.ts`。

---

## 根因：幽灵引用

`showModelPopup()` 的开头是这样写的：

```typescript
export function showModelPopup(): void {
    dom.sceneOverlay.innerHTML = "";
    dom.sceneOverlay.classList.add("overlay-model");
    dom.sceneOverlay.dataset.popupType = "model";

    if (!stackRegistry.modelStack) {
        stackRegistry.modelStack = makeModelStack();
    }

    stackRegistry.modelStack.reset(buildModelRoot());
}
```

注意这一行：

```typescript
stackRegistry.modelStack.reset(buildModelRoot());
```

`reset()` 是 `SlideMenu` 的方法，它会重新渲染菜单内容。但 `SlideMenu` 实例是在第一次打开弹窗时创建的，之后一直存在。

问题在这里——

`innerHTML = ""` 把 `sceneOverlay` 内部的所有 DOM 节点都销毁了。但 `stackRegistry.modelStack` 这个 `SlideMenu` 实例内部，仍然持有那些**已被销毁的 DOM 节点引用**。`reset()` 调用时，`SlideMenu` 试图操作这些已分离的节点——内容实际上没有被渲染到 `sceneOverlay`。

看起来「打不开」，实则是**内容从未被挂载**。

Riku 用一个比喻向 Jieling 解释：

> 你把房间里的家具全扔了（`innerHTML = ""`），但管家（`modelStack`）手里还拿着旧家具的购买发票，每次你说「重新布置」，他都对着发票念念有词，但房间里依然空空如也——因为他操作的家具早已不存在了。

---

## 修复：每次重建

修复方案很直接——每次 `showXxx()` 都 `new SlideMenu()`，不再复用旧实例。

```typescript
// 修复前
if (!stackRegistry.modelStack) {
    stackRegistry.modelStack = makeModelStack();
}
stackRegistry.modelStack.reset(buildModelRoot());

// 修复后
stackRegistry.modelStack = makeModelStack();
stackRegistry.modelStack.reset(buildModelRoot());
```

五个弹窗函数都做了同样的修改：

| 文件 | 函数 | 状态 |
|------|------|------|
| `library-core.ts` | `showModelPopup()` | ✅ 每次重建 `modelStack` |
| `motion-popup.ts` | `showMotionPopup()` | ✅ 每次重建 `motionStack` |
| `settings.ts` | `showSettings()` | ✅ 每次重建 `settingsStack` |
| `scene-menu.ts` | `showSceneMenu()` | ✅ 每次重建 `sceneStack` |
| `env-menu.ts` | `showEnvMenu()` | ✅ 每次重建 `envStack` |

## 同步修复

在追踪这个问题的过程中，Riku 还发现了几个相关的缺陷：

### 修复 A：`overlay-fade-out` 残留

`toggleOverlay` 的 cross-fade 分支会给旧弹窗加上 `overlay-fade-out` 类，让它渐隐。但如果用户按 ESC 键或点击画布中途关闭，`overlay-fade-out` 类可能残留在 `sceneOverlay` 上。下次打开时，弹窗内容存在，但被 `overlay-fade-out` 的 `opacity: 0` 隐藏了——看起来又是「打不开」。

修复：`closeAllOverlays()` 中添加清理：

```typescript
export function closeAllOverlays(): void {
    document.querySelectorAll<HTMLElement>("[data-overlay].visible").forEach(el => {
        el.classList.remove("visible", "overlay-fade-out"); // ← 新增
    });
    // ...
}
```

### 修复 B：画布点击只关闭，不恢复

之前，`_toggleOverlays()` 实现了「点击画布关闭弹窗，再次点击恢复弹窗」的逻辑。这需要维护 `_lastHidden` 数组和 `_overlaysHiddenByClick` 标志，状态机变得复杂。

Jieling 决定简化：**画布点击只有关闭权，开权归按钮独享**。这符合 DanceXR 的法理，也消除了状态不同步的可能。

```typescript
function _toggleOverlays(): void {
    const all = _getAllOverlays();
    const anyVisible = all.some(el => el.classList.contains("visible"));

    if (anyVisible) {
        all.forEach(el => el.classList.remove("visible"));
        setPopupOpen(false);
        syncNavAriaExpanded();
    }
    // 不再有「恢复」分支
}
```

## 四轮迭代

这个问题从首次报告到最终修复，历经四轮迭代：

| 轮次 | 诊断 | 修复 | 结果 |
|--------|------|------|------|
| 1 | toggle 行为不一致 | 统一容器到 `sceneOverlay` | 引入新 bug |
| 2 | 关闭后打不开 | 修复 `syncNavAriaExpanded()` 位置 | 未根治 |
| 3 | `overlay-fade-out` 残留 | `closeAllOverlays()` 添加清理 | 未根治 |
| 4 | **DOM 引用悬空**（根因） | 五个 `showXxx()` 改为每次 `new SlideMenu()` | ✅ 验证通过 |

Riku 在提交信息里写：

> 四轮迭代才找到根因。教训：当「打不开」伴随「aria-expanded 正常」时，问题不在状态机，而在 DOM 是否真的被挂载。

## 尾注

修复完成后，Riku 在 `.workbuddy/memory/` 里记了一笔：

> **经验教训**：`innerHTML = ""` 是毁灭性操作。任何持有该容器内部 DOM 引用的对象，在该操作之后都变成了「幽灵引用」。要么在 `innerHTML = ""` 后手动清除所有引用（容易漏），要么每次重建对象（简单但可能牺牲一点性能）。对于弹窗这类低频次 UI，重建的成本可以忽略，安全的价值远高于那几毫秒。
