## 二、透明与卡片

王逸提到了一个名字——DanceXR。

「他们是用透明框加上不透明的卡片，实现组件之间的隔离效果。」

Sisyphus 想象了一下那个画面。整个弹窗的底色消失，每个菜单项变成一块独立的半透黑卡片，悬浮在 3D 场景之上。卡片之间有 3px 的间隙，透过间隙能看到模型的裙摆在旋转。

「试试，」王逸说。

这一试就是三小时。

`.overlay` 的 `background` 被设置为 `transparent`。
`.slide-item` 获得了 `--card-bg: rgba(0,0,0,0.5)`。
`.lcard` 作为分组容器，包裹相关按钮。
`.render-card` 从所有用 `cardContainer` 的地方被移除——不再叠层。

王逸看着效果：「不对，三层颜色叠在一起了。」

Sisyphus 排查。`render-card` → `lcard` → `slide-item`，三层 50% 黑叠加，有效透明度叠到 87.5%。这是三层玻璃看世界。

「每个行元素只带一层底色，」王逸拍板，「`lcard` 提供背景，行内元素透明。」

于是标准定了下来：

```
overlay  rgba(0,0,0,0.15)   ← 弹窗底景
  └─ lcard  --card-bg       ← 卡片组，包裹相关按钮
       └─ slide-item  透明   ← hover 变亮
       └─ cs-row      透明
```

## 三、CSS 变量战争

「变量名以功能性为主，防止随意使用大、小字体，」王逸说。

Sisyphus 看着现有的 `--font-size-md`、`--font-size-lg` 陷入了沉思。这些名字的问题在于：它们是**尺寸语义**而不是**功能语义**。开发者在写代码时看到 `lg` 就会想：「这条文字大一点应该用 `lg` 吧？」于是本该是辅助说明的文字用上了标题的大小。

改。

```
--font-ui      13px   菜单行、按钮、列表
--font-ui-sm   11px   辅助说明、面包屑
--font-ui-xs   10px   极小标志、徽章
--font-title   14px   弹窗标题、卡片标题
--font-title-lg 16px  空状态、强调文字
--font-time    12px   时间数字
```

「契约：写 `font-size: var(--font-xxx)` 时，变量名必须匹配元素角色。review 时看到 `.menu-row { font-size: var(--font-title) }` 一眼就错。」

然后是缩放。王逸想要一个全局 UI 缩放功能——高分屏调大、低分屏调小，一个滑块联动全部。

`--ui-scale: 1`

所有 `--font-*` 改为 `calc(13px * var(--ui-scale))`，`--popup-width` 改为 `calc(280px * var(--ui-scale))`。

「调一个变量，全 UI 跟着走，」Sisyphus 想，「零侵入，这才叫设计。」

## 四、Go 端的远征

「设置改了要记住，下次启动还在。」

这要求跨过 Go/TypeScript 的边界。

`app.go` 新增 `UIState` struct：

```go
type UIState struct {
    Scale      float64 `json:"scale"`
    PopupWidth int     `json:"popupWidth"`
    Accent     string  `json:"accent"`
    FontFamily string  `json:"fontFamily"`
    Animations bool    `json:"animations"`
    BlurBg     bool    `json:"blurBg"`
}
```

`wails generate module` 生成前端 binding。前端启动时 `GetConfig` → `restoreUIState` → 写 `:root` 变量。

「等等，」王逸打断，「`SetUIState({ scale: v })` 会覆盖其他字段为零值。」

Sisyphus 脊背一凉。确实——Go 反序列化缺失字段为 0 和 ""，然后整个 UIState 被替换。

拆。四个独立 setter：`SetUIScale`、`SetUIPopupWidth`、`SetUIAccent`、`SetUIFontFamily`，遵循代码库已有模式。

字体存储也踩了坑——直接存 CSS 值字符串「格式变了存的数据就失效」。改为存 key（`system`/`noto`/`yahei`），前端用 `FONT_MAP` 解析。

## 六、场景菜单的卡片化

「你看到没有，」王逸指着屏幕，「灯光面板里方向光色和环境光色是两个独立的颜色选择器，但它们被挤在一个 render-card 里，中间还夹着分隔线。」

Sisyphus 看了看。cs-rows、clr-block、addToggleRow——三种不同的组件混在一个容器里，一条 `height:1px` 的分隔线试图把它们分开。

「用卡片分组。」

`cardWrap` 应运而生——一个简单的函数，创建一个 `lcard` div，把内容放进去，然后移除容器上的 `render-card` 类：

```typescript
function cardContainer(container, fn) {
    container.classList.remove("render-card");
    const card = document.createElement("div");
    card.className = "lcard";
    fn(card);
    container.appendChild(card);
}
```

场景根菜单被分成了 4 个卡片：

```
场景管理 —— 预设场景 / 保存场景 / 加载场景
渲染与相机 —— 相机模式 / 灯光 / 渲染 / 物理
程序化动作
工具 —— 截图
```

每个卡片独立悬浮，间 6px。分隔线消失了——卡片的边界就是最好的分隔。

## 七、模型库 2481 行的崩塌

「library.ts 2481 行了，」王逸说，「得拆。」

Sisyphus 早就想拆了。`library.ts` 混合了三个职责域：模型库核心逻辑、9 个模型详情子菜单、动作弹窗和舞蹈套装。更可怕的是，`sceneStack` 被 `buildModelDetailLevel` 引用，但没有 import——靠 Vite 打包时的全局作用域「碰巧」工作。

注册表模式。

一个 `stackRegistry` 对象，由 `library-core.ts` 持有，子模块通过它间接引用 `modelStack` 和 `sceneStack`。

```
library.ts          ← 入口，13 行，re-export
  ├── library-core.ts  ← 模型库核心 + Stack 持有者 + 注册表
  ├── model-detail.ts  ← 模型详情 9 个子菜单
  └── motion-popup.ts  ← 动作弹窗 + 舞蹈套装
```

提取 `slideRow`、`addSliderRow`、`addToggleRow` 到 `ui-helpers.ts`——三份重复代码合并为一份。

当 Sisyphus 把 2481 行的 `library.ts` 拆成 4 个文件、总行数反而降低了 200 行时，他感到了一种代码上的救赎。
