## 📊 Wails v3 Events System Analysis

### **Event System Type**: `app.Event` Manager API (Custom Events System)

### **Core Architecture**
Wails v3 introduces a **unified event system** for pub/sub communication between Go backend and JavaScript frontend. The system provides bidirectional event communication with type safety and proper event objects.

## 📡 3. Event Communication Patterns

### **Pattern 1: Go → JavaScript (Backend to Frontend)**

**Go Backend Code**:
```go
// Emit custom event from Go
app.Event.Emit("model-loaded", map[string]interface{}{
    "path": "/path/to/model.pmx",
    "name": "Hatsune Miku",
    "vertices": 12345,
})

// Emit application event
app.Event.EmitApplicationEvent(events.Common.ThemeChanged, themeData)
```

**JavaScript Frontend Code**:
```typescript
import { Events } from '@wailsio/runtime';

// Register event listener
const unsubscribe = Events.On('model-loaded', (e: CustomEvent) => {
  console.log('Model loaded:', e.data);
  // e.data contains: { path, name, vertices }
});

// Unsubscribe later
unsubscribe();
```

---

### **Pattern 2: JavaScript → Go (Frontend to Backend)**

**JavaScript Frontend Code**:
```typescript
import { Events } from '@wailsio/runtime';

// Emit event from frontend
Events.Emit('ui-action', { action: 'play-animation', modelId: 'miku-001' });
```

**Go Backend Code**:
```go
// Register event listener in Go service
app.Event.On("ui-action", func(e *application.CustomEvent) {
    action := e.Data.(map[string]interface{})
    log.Printf("UI action received: %s for model %s",
        action["action"], action["modelId"])
})
```

---

### **Pattern 3: Window-to-Window Communication**

```go
// Emit event to all windows
app.Event.Emit("theme-changed", themeData)

// Emit event to specific window
window, _ := app.Window.GetByName("editor")
window.EmitEvent("model-updated", modelData)
```

## 🎛️ 4. Event Registration & Type Safety

### **Event Registration (Go init phase)**:

```go
package main

import (
    "github.com/wailsapp/wails/v3/pkg/application"
)

func init() {
    // Register event with expected data type
    application.RegisterEvent[ModelLoadedData]("model-loaded")

    // Register void event (no data)
    application.RegisterEvent[application.Void]("model-cleared")
}

type ModelLoadedData struct {
    Path     string `json:"path"`
    Name     string `json:"name"`
    Vertices int    `json:"vertices"`
}
```

### **Type-Safe Emission**:

```go
// With type registration - data is type-checked at runtime
app.Event.Emit("model-loaded", ModelLoadedData{
    Path: "/models/miku.pmx",
    Name: "Hatsune Miku",
    Vertices: 12345,
})

// Returns boolean indicating if event was cancelled
cancelled := app.Event.Emit("critical-action", data)
if cancelled {
    log.Println("Event was cancelled by a hook")
}
```

## 🔄 5. Event Lifecycle Management

### **Registering Listeners**:

```go
// Single listener
cancelFunc := app.Event.On("event-name", func(e *application.CustomEvent) {
    // Handle event
    log.Printf("Event %s received with data: %v", e.Name, e.Data)
})

// Multiple listeners (returns slice of cancel functions)
cancelFuncs := app.Event.OnMultiple("event-name", []application.EventListener{
    listener1,
    listener2,
    listener3,
})

// Application events
app.Event.OnApplicationEvent(events.Common.ThemeChanged, func(e *ApplicationEvent) {
    // Handle system theme change
    applyTheme(e.Data.(ThemeData))
})
```

### **Removing Listeners**:

```go
// Remove specific listener
app.Event.Off("event-name")

// Remove all listeners for event
app.Event.Off("event-name")

// Reset all events (remove all listeners)
app.Event.Reset()
```

## 📊 6. Key Features & Benefits

### **Type Safety**:
- ✅ Events can be registered with specific data types
- ✅ TypeScript IDEs provide autocompletion
- ✅ Data validation at build time (frontend) and runtime (Go)

### **Cancellation Support**:
```go
// Listener can cancel event propagation
app.Event.On("critical-action", func(e *application.CustomEvent) {
    if !userHasPermission(e.Data.(ActionData)) {
        e.Cancel() // Prevent further propagation
    }
})
```

### **Metadata & Debugging**:
- ✅ Event objects contain `name`, `data`, `cancelled` fields
- ✅ Source window tracking
- ✅ Better debugging experience

### **No Context Required**:
- ✅ Unlike v2, v3 doesn't require threading context
- ✅ Direct access via `app.Event` methods

### **Application Events**:

## ⚠️ 9. Important Notes & Best Practices

### **Event Naming Convention**:
- Use kebab-case: `"model-loaded"`, `"theme-changed"`, `"ui-action"`
- Be specific and descriptive
- Include domain prefix for large apps: `"models:loaded"`, `"editor:selection-changed"`

### **Data Serialization**:
- ✅ Structs with JSON tags work best
- ❌ Avoid complex types that don't serialize well
- 📦 Keep data payloads small (< 1MB recommended)

### **Error Handling**:
```go
// In event listener
app.Event.On("risky-operation", func(e *application.CustomEvent) {
    defer func() {
        if r := recover(); r != nil {
            log.Errorf("Event handler panic: %v", r)
        }
    }()

    // Safe operation
})
```

### **Memory Management**:
- Always store unsubscribe functions if you need to remove listeners
- Clean up listeners in cleanup/dispose functions
- Avoid memory leaks with long-lived listeners

## 🔗 10. Migration from Wails v2

### **v2 → v3 Changes**:

| v2 | v3 |
|----|----|
| `runtime.EventsOn(ctx, "event", handler)` | `Events.On("event", handler)` |
| `runtime.EventsEmit(ctx, "event", data)` | `Events.Emit("event", data)` |
| `*application.WailsEvent` | `*application.CustomEvent` |
| Context required | No context needed |
| Variadic parameters | Type-safe event objects |
| `app.Events.On()` | `app.Event.On()` (singular) |

### **Example Migration**:
```typescript
// v2
import { EventsOn, EventsEmit } from '../wailsjs/runtime/runtime';
EventsOn("update", (data) => { console.log(data) });
EventsEmit("action", data);

// v3
import { Events } from '@wailsio/runtime';
Events.On("update", (data) => { console.log(data) });
Events.Emit("action", data);
```
