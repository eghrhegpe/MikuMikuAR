本仓库采用「前后端各自轻量、无第三方依赖」的日志方案，未引入集中式日志框架或结构化日志后端。

## Go 后端（Wails 服务层）

- 使用标准库 `log/slog`，在 `internal/app/app.go` 中封装了三个私有方法 `safeLogInfo / safeLogWarning / safeLogError`，仅在 `wailsApp` 可用时调用，避免初始化阶段 panic。
- 所有业务模块直接通过这三个方法输出 Info/Warning/Error 级别日志，未自定义 Handler、未设置 JSON 格式、未附加结构化字段（如 module、traceId），属于最简控制台输出。
- 未见全局 slog 配置或 sink 注册逻辑，日志默认输出到进程 stdout/stderr。

## TypeScript 前端（Wails + Babylon.js）

### 统一入口

- 统一入口为 `frontend/src/core/logger.ts`，从 ADR-141 拆分自 `utils.ts`，提供 `logInfo / logWarn / logError` 三个函数，统一以 `[tag] message` 前缀包装后走 `console.info/warn/error`。
- 全项目约定：所有模块必须从此文件 import，禁止再直接从 `utils.ts` 导入旧版日志函数，以避免循环依赖。

### 使用覆盖范围

当前 25+ 模块通过此统一接口输出日志，涵盖：

| 模块类别 | 代表文件 | 典型 tag |
|---------|---------|---------|
| 场景核心 | `scene.ts`, `camera.ts`, `render-loop.ts` | `'scene'`, `'camera'` |
| 环境子系统 | `env-sky.ts`, `env-water.ts`, `env-ground.ts`, `env-particles.ts` | `'env'` |
| 渲染 | `renderer.ts` | `'render'` |
| 动作/感知 | `proc-motion-bridge.ts`, `perception.ts`, `feet-adjustment.ts`, `beat-detector.ts` | `'motion'`, `'perception'` |
| 换装 | `outfit-overlay.ts`, `audio.ts` | `'outfit'` |
| 菜单/UI | `menu.ts`, `model-detail.ts`, `library-setup.ts`, `settings-*.ts` | `'menu'`, `'library'` |
| 基础设施 | `safe-call.ts`, `shortcut-registry.ts` | `'safeCall'`, `'shortcut'` |

### 安全调用三件套（ADR-146）

`frontend/src/core/safe-call.ts` 封装了「吞错并 logWarn」的通用模式，替代大量手写 `try/catch + logWarn` 重复：

```typescript
import { safeCall, safeCallVoid, safeCallAsync } from '@/core/safe-call';

// 同步有返回值：异常返回 undefined
const v = safeCall('audio', 'decode', () => decode(buf));

// 同步无返回值：异常仅 logWarn
safeCallVoid('physics', 'step', () => step(dt));

// 异步：Promise<T | undefined>，不传播 rejection
const data = await safeCallAsync('init', 'Library init', () => initLibrary());
```

与 `utils.ts` 的 `swallowError` 区别：`swallowError` 固定 tag `'swallow'` 且丢失原始上下文；`safe-call` 三件套保留调用方传入的 tag/msg，便于按模块聚合排查。

## 设计决策与约束

- 前后端均未接入集中式日志收集（如 ELK、Sentry、Wails 内置 Logger API 未被使用），定位为开发期调试输出。
- 前端 logger 刻意保持零依赖、无状态，不引入任何运行时开销，适合高频渲染路径旁路记录。
- `safe-call` 统一了「吞错」模式，避免散落的 `try/catch` 遗漏 logWarn 或静默吞错。
- 若需升级至结构化日志，Go 侧应配置 `slog.NewJSONHandler` 并注入全局 `slog.SetDefault`；前端可考虑将 `console.*` 替换为 Wails 提供的 `application.Logger`（bindings 已生成类型）。

## 开发者规则

1. **所有新模块**必须从 `@/core/logger` 导入 `logInfo/logWarn/logError`，禁止使用裸 `console.*`（除非在 logger.ts 本身内部）。
2. **tag 命名**：使用模块短名（如 `'env'`、`'scene'`、`'outfit'`），保持全局唯一，便于按标签过滤。
3. **异常处理**：非关键路径的 catch 块统一使用 `safeCall/safeCallVoid/safeCallAsync`，不要手写 `try { } catch { logWarn(...) }`。
4. **关键路径**（如模型加载、配置持久化）的异常必须同时通知用户（`setStatus(t('xxx.failed'), false)`），不能仅 logWarn。
5. **禁止**：不要在渲染循环（`scene.registerBeforeRender`）内输出 `logInfo`，避免性能退化；warn/error 可酌情使用。
