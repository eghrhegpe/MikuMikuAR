本仓库在 Go 后端与 TypeScript 前端之间采用「结构化错误信封 + 哨兵文本」的跨桥错误处理方案，统一了用户可见错误的国际化、可观测性与容错。

## 体系概览

- **Go 端**：通过 `internal/i18nerr` 定义面向用户的错误类型 `UserError`，携带 i18n code 与占位参数；通过 `internal/util/errors.go` 提供 `WrapError`/`WrapErrorf` 统一错误链包装；通过 `internal/util/safecall.go` 提供 panic→error 的安全调用封装。
- **Wails 跨桥约束**：Wails v3 将 Go error 以纯文本形式序列化到前端（`messageprocessor_call.go` → `transport_http.go` → `runtime.js throw new Error(await response.text())`），无法直接传递结构化对象。因此 `UserError.Error()` 把 JSON 信封嵌入 `.Error()` 字符串末尾，由前端解析。
- **前端**：`frontend/src/core/i18n/goerr.ts` 的 `translateGoError` 按哨兵 `@@GOERR@@` 提取信封，用 `t('goerr.<code>', params)` 翻译，缺失 key 时回退信封内中文 msg，再失败则回退原始文本。

## 架构约定

1. **错误分层**
   - 内部错误：使用 `fmt.Errorf` + `%w` 包装，配合 `util.WrapError(op, err)` 添加操作上下文，保持错误链可追溯。
   - 用户可见错误：一律通过 `i18nerr.New(code, fallbackMsg, params...)` 构造，禁止直接 `fmt.Errorf` 中文文本暴露给前端。
2. **信封契约**
   - Go 端 `UserError.Error()` 输出格式为 `<可读msg>\n@@GOERR@@{"code":"...","params":{...},"msg":"..."}`。
   - 前端仅按最后一个 `@@GOERR@@` 截取尾部 JSON，忽略 Wails 前缀包裹文本。
   - 若前端 i18n key 缺失，回退到信封内的 `msg`（中文兜底），再失败回退原始错误文本。
3. **panic 防护**
   - 所有可能触发 panic 的 I/O、解压、WASM 调用等，必须通过 `util.SafeCall` 或 `util.SafeCallVoid` 包裹，panic 值会被转换为带堆栈的错误，且对 `errors.Is` 透明。
4. **平台差异错误**
   - Android 不支持的功能（如内容 URI、外部程序启动、文件系统监听）统一返回 `i18nerr.New("*.androidNotSupported", ...)`，前端自动翻译。

## 开发者规则

- 对外暴露给用户的信息一律走 `i18nerr.New`，code 遵循 `模块.子域.具体错误` 命名空间（如 `software.notFound`、`screenshot.dirCreateFailed`）。
- 内部调试日志可使用 `fmt.Errorf`，但向上层传播时必须用 `util.WrapError` 保留链。
- 任何可能 panic 的函数调用必须包一层 `SafeCall`，不要在业务逻辑中手写 `defer recover`。
- 新增 i18n code 后，需在对应语言包（`frontend/src/core/i18n/locales/*.ts`）补充 `goerr.<code>` 键值，否则前端会回退到中文兜底。
- 测试应覆盖信封解析路径（参考 `frontend/src/__tests__/goerr.test.ts`），确保新 code 被正确翻译。
